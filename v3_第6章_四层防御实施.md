# 第 6 章 四层防御实施

---

## 6.0 四层总览:从核心矩阵看四层职责

第 3 章把 framework 的核心矩阵定义为「4 层防御 × CIK 三维」的二维结构——横轴是按工具调用时间组织的四层防御,纵轴是按属性组织的 CIK 三维。本章逐层展开这四层防御的具体机制,把第 4 章建立的攻击面坐标、第 5 章列出的威胁场景翻译为可工程化的防护手段。

四层防御各自承担明确职责。**事前准入层**(pre-admission)在智能体或 Skill 上线之前完成"门槛筛查"——供应链扫描、身份配置基线、部署管线审核;它的失效后果不是即时副作用,而是把不应进入系统的实体放进了运行环境,后续每一层的防护都建立在"事前已准入实体"这一前提之上。**事中拦截层**(in-line interception)位于工具调用执行路径上,是实时拦截主战场——预执行防火墙、策略引擎、能力凭证、沙箱与人工审批共同完成"在副作用发生之前完成判决"的工程目标;失守即不可逆。**事后运营层**(post-execution operations)在调用完成之后接管——防篡改审计链记录证据,五维智能体画像揭示长期状态偏移,信任评分(Trust Decay)在异常累积时主动收缩权限。**跨智能体层**(cross-agent)处理多智能体协同特有的信任问题——身份冒充、消息注入、流程操纵、共享知识库污染。

下图把核心矩阵 12 个单元格在四层之间的分布做了可视化:

![核心矩阵 12 单元格 + 主流防护方案的覆盖叠加](figures/v3/overview.png)

四层并非互相独立。一次工具调用的完整生命周期会**穿过四层**——Skill 在事前准入层通过扫描进入注册表,在事中拦截层接受策略引擎判决,在事后运营层留下审计与画像数据,在跨智能体协同时其身份与信任关系受跨智能体层管控。本章前四节按层展开各自的机制,§6.5 专门论述层间协同与缝隙,回应第 3 章顶层原则中"统一跨层策略执行"在工程上的落地形式。

## 6.1 事前准入层:供应链与配置门槛

### 6.1.1 层职责定位

事前准入层的核心问题是:**这个智能体或这个 Skill 可以上线吗?** 这一问题落在工具调用发生之前,涉及三类对象——Skill(对应 C 维度)、身份配置(对应 I 维度)、知识基线(对应 K 维度)。本层的工程价值不在拦截某次调用,而在压缩攻击面——若 Skill 在准入阶段未通过扫描,整个事中拦截层无需为这个 Skill 应对的威胁负责。

第 5 章 §5.1.5 列出的供应链信任升级类攻击(AT5)主要在这一层应对。R12 Microsoft AIRT 失效模式分类法识别的「部署管线投毒」(Provisioning Poisoning,即在 Skill 注册或部署管线环节注入恶意构件)对应 S11 攻击面,这一攻击面的全部生命周期都发生在事前层——它是事中层无法补救的盲区。

### 6.1.2 Skill 供应链安全:扫描器与 CVE 情报

Skill 是智能体执行能力的载体。一个 Skill 既可能是开源 npm 包、PyPI 模块、MCP 工具,也可能是企业内部维护的脚本集合。无论形式如何,Skill 在被智能体引用之前必须经过两类筛查:**扫描器筛查**与**情报订阅筛查**。

扫描器的工作对象是 Skill 的代码与元数据。R4 Cisco DefenseClaw 提供了较为完整的 Skill 扫描参考实现——它在 Skill 注册到智能体框架之前自动执行四类检查:静态代码分析(寻找已知漏洞模式)、依赖审计(逐项核对依赖包的 CVE 状态)、签名与发布者验证(确认 Skill 出自受信任发布者)、行为模式画像(执行一次受控的样本运行,记录其工具调用与网络请求模式)。其中第四项行为模式画像是 DefenseClaw 相对其他扫描器的差异化能力——它不仅判断 Skill"有没有已知漏洞",还判断 Skill"运行起来像不像它声称的样子"。

情报订阅与扫描器形成互补关系。R18 Knownsec OpenClaw Security 把 GitHub Advisory Database 中已收录的 245 个 OpenClaw 相关漏洞通过 RSS / webhook 推送到企业 SIEM,事中层的策略引擎在收到新增 CVE 通知后立即对所有受影响 Skill 执行回查,触发 Skill 失效或版本回滚。这一设计的关键在于:**Skill 准入并不是一次性事件**——一个昨天通过扫描的 Skill,今天可能因为上游披露的新 CVE 而需要重新评估。R20 OpenClaw CVE Tracker 提供了 CVE 增长曲线的实证——智能体相关 CVE 的披露速度远快于补丁就绪速度,情报订阅是事前准入层与事后运营层之间的实时反馈链。

R5 Microsoft Agent Governance Toolkit 把上述两类筛查打包为统一的「智能体注册中心」——所有 Skill 必须先在注册中心登记并通过扫描,智能体运行时只能引用注册中心已审核的 Skill。这种"白名单 + 注册中心"模式比纯扫描器更适合企业部署场景,因为它把 Skill 的访问控制问题统一到注册中心的权限管理中。

### 6.1.3 部署管线审核:S11 攻击面的专属应对

部署管线审核是 v3 framework 在 v2 基础上加入的新内容,直接回应 R12 AIRT 提出的 Provisioning Poisoning 失效模式。所谓部署管线投毒,是攻击者在智能体或 Skill 的构建、签名、分发环节注入恶意载荷,使得最终落到生产环境的构件已经被污染。这类攻击在传统 OWASP 与 MITRE 分类法中并未单独命名,但对智能体生态尤为关键——智能体的能力依赖大量第三方 Skill,而这些 Skill 的构建管线分散在各发布者手中。

部署管线审核的核心工程实践有三项:**构建过程可重现验证**(reproducible builds)——同一份源码在两个独立环境构建出的产物应当字节级一致,任何不一致即代表构建链遭篡改;**签名链可追溯**——每个 Skill 必须由可识别的发布者签名,签名验证失败即拒绝准入;**部署清单完整性**——记录从源码到构件的每一步依赖与变更,在事后审计层(§6.3)被防篡改审计链持续监控。R5 AGT 与 R4 DefenseClaw 在这三项实践上提供了相对完整的工程参考。

### 6.1.4 身份配置基线与知识基线

事前准入层的 I 维度工作是身份配置基线——智能体的初始身份、可访问角色、子智能体派生关系在部署时一次性确认并固化。第 4 章 §4.4.3 描述的「身份冒充」攻击(AT1 + R12 Agent Impersonation)需要在事前层先建立"什么是合法身份"的基线,事后层的"信任异常使用检测"才有比对的对象。

K 维度工作是知识基线——系统提示词在部署时签名,RAG 知识库在准入时建立内容指纹与白名单,长期记忆模块的初始状态被固化为可比对的基线。这一基线是事后层「记忆隐患检测」的对照——若不知道知识应当是什么样,就无法判断它是否被污染。

### 6.1.5 硬件可信根: TEE 与 Remote Attestation

软件层防御 (策略引擎 / 沙箱 / prompt 过滤) 在敌对运维 (adversarial ops) 威胁模型下系统性失效——root、hypervisor 与 cloud operator 可绕过一切应用层管控; R32 论证硬件 Trusted Execution Environment (TEE) 与 Remote Attestation (RA) 是 P4 可验证原则 (第 3 章 §3.2.4) 在 high-stakes 部署中的**硬件实现层**, 是事前准入必须保留的工程接入点。

R32 给出的威胁模型升级具有结构意义: agent 运行期持续累积敏感上下文——用户凭证、企业知识切片、API token、跨步推理中间状态——在内存中明文驻留远长于传统请求-响应服务; 一旦特权对手触达宿主, §6.1.4 身份配置与知识基线、§6.4.2 R30-N1 一级代理身份均无应用层抓手。硬件可信根把信任锚从"进程自证"下沉到"硬件厂商签发的度量", 是 cross/I 单元在硬件层的延伸。

主流 TEE 平台在隔离粒度与 agent 适配性上差异显著, 须按部署形态选型:

| 平台 | 类型 | 关键约束或特性 |
|---|---|---|
| Intel SGX | CPU enclave (细粒度) | enclave 内存上限制约 >30B 模型部署 |
| Intel TDX | VM-level CC | 全 VM 隔离, 适合大模型 inference container |
| AMD SEV-SNP | VM-level CC | 内存加密 + nested page table 完整性, 约 209× CPU 周期开销 |
| ARM TrustZone | 安全/普通 World 分区 | 移动与边缘 agent 部署主流 |
| ARM CCA | Realm (TrustZone 下一代) | 支持 confidential VM, 服务器/边缘统一栈 |
| NVIDIA H100 CC | GPU TEE | 首个生产级 GPU TEE, 4-8% 开销, 使生产 LLM inference 可行 |

代表性系统覆盖 inference 切分、协议扩展与容器化路径: TEESlice、CMIF、TEECHAT、Omega、**AttesMCP** (Model Context Protocol 的 attestation 扩展, 与 §6.4.2 / §6.4.9 直接关联)、CAEC (共享内存减 16.6-28.3% 足迹)、BlockA2A。AttesMCP 是当前 MCP 生态唯一把 TEE 度量嵌入 tool-server 握手的方案, 必须纳入 MCP 安全演进观察项。

Remote Attestation 把 "agent 自称是谁" 升级为 "TEE 担保 agent 在何种代码与配置下运行"——RA 报告携带 measurement、平台 quote 与 nonce, verifier 比对策略基线后放行。在多代理链路中, RA 是 §6.4.2 R30-N1 一级身份的硬件锚点, 也是 §6.4.8 R30-N7 re-attestation 的合法触发条件: 代码或配置漂移即 measurement 变化, attestation 失败即终止下游授权传播, 自然实现"漂移即撤权"。

必须诚实声明边界——TEE **必要但不充分**: 它不解决 LLM 输出语义完整性、不阻断内容级 prompt injection、不保证多代理认证新鲜性 (freshness)、无法消除微架构侧信道泄漏。TEE 仅为应用层防御提供"未被特权对手篡改"的执行底座, 仍须与 R5 AGT 应用层隔离、R22 CaMeL 控制流-数据流分离、协议级新鲜性机制叠加。把 TEE 理解为银弹既违背 R32 原意, 也会在金融、医疗、政府、国防场景 (详见第 9 章 §9.2) 造成误判。R32 在 v3.1 升核心评估中维持 survey 定位, **不升核心论据列表** (附录 C §C.2), 工程价值通过本节、§6.4.2 与 Ch9 §9.2 三处结构补强承载。

### 6.1.6 Stochastic Supply Chain: 运行时依赖与 MitE 威胁模型

R35 Supply Chain SOK(Yang 等,arXiv:2602.19555,ICLR 2026 Workshop)论证 agent 系统的供应链不止 build-time(静态依赖)而包含 **inference-time(动态依赖)** 一极——后者通过 LLM 推理在调用现场"动态选定",事前无法穷举;任何沿用 SBOM(Software Bill of Materials) + 静态扫描的供应链防御在 dynamic 形态下出现系统性失配。本节给出事前准入层对运行时供应链的工程接入点:既承认 §6.1.2 / §6.1.3 的静态扫描与管线审核仍是必要门槛,也明确这些门槛在 runtime 维度的盲区,并指出与之配套的 Zero-Trust Runtime Architecture 接入位置。

**静态 vs 动态供应链的范式区分**(R35 §3 Table 1):

| 维度 | Static Supply Chain | Stochastic / Dynamic Supply Chain |
|---|---|---|
| Resolution Time | Build / Deploy Time | **Inference / Runtime** |
| Dependency Type | Libraries / Binaries | **Data Context, APIs, Tools** |
| Topology | Directed Acyclic(DAG) | **Cyclic(Feedback Loops)** |
| Attack Surface | Code Vulnerabilities | **Semantic Manipulation** |
| Upstream Source | Verified Vendor | **Runtime Information Sources** |

该表的工程含义是:**静态供应链的全部假设——可枚举、可签名、可一次性审计——在动态供应链一极均不成立**。一个 agent 在 runtime 检索到的 wiki 条目、forum 帖子、MCP 工具描述,既不出现在任何 SBOM 中,也不带可验证发布者签名,却被 agent 主动升格为决策依据。沿用 build-time 工具(snyk、trivy、dependabot 等)既检测不到这类依赖的存在,更无法判定它们是否被污染。

**MitE(Man-in-the-Environment)威胁模型**。R35 命名的 MitE 与传统 MitM(Man-in-the-Middle)形成对仗——MitM 攻陷信道,MitE **不攻陷信道、不改模型权重、不动 system prompt**,而是污染 agent 所依赖的环境(wiki 条目、代码仓库 README、论坛回复、tool registry 元数据)。环境是 agent 的 *primary source of truth*,agent 主动检索环境内容并把它升格为指令——R35 把这一现象命名为 **Prompt-Data Isomorphism**(提示词与数据在 LLM 处理路径上同形),它从根本上解释了为什么"过滤外部输入"在 agent 形态下不再是充分防御。MitE 与本章既有子节构成 build/runtime 对偶关系:§6.1.2 的 Skill 供应链投毒攻击的是 **build-time 注册表**,§6.1.3 的部署管线投毒(R12 Provisioning Poisoning)攻击的是 **build-time 分发链**,而 MitE 攻击的是 **runtime 信息源**——同一 AT5 对抗技术(供应链信任,见第 5 章 §5.1.5)在不同时序展开的两个面。

**工具供应链 3 阶段攻击**(R35 §4 分类)。运行时工具供应链按 agent 与工具交互的时序可细分为三个阶段,每阶段独立产生攻击窗口:① **Discovery**——agent 在 runtime 检索可用工具(MCP server 列表、tool registry 查询、agent-card 拉取),恶意条目可通过 SEO-style 投毒进入候选集;② **Implementation**——同名工具的实现可被静默替换或版本切换,agent 仍以为在调用昨日扫描通过的版本;③ **Invocation**——工具调用参数在生成路径上被 in-context payload 污染,合法工具被诱导执行越权操作。这三阶段与 ASF 攻击面 **S2(技能分发)、S5(工具调度)、S11(部署管线)** 正交但侧重 runtime——S2/S11 在 build-time 已被 §6.1.2 / §6.1.3 覆盖,Discovery 与 Implementation 的 runtime 形态则必须由本节描述的事前准入策略接管:对所有 runtime 可发现的工具源(MCP endpoint、agent-card URL、registry domain)预先建立白名单与发布者签名校验,任何 runtime 新出现的工具源默认拒绝准入并降级为人工复核。

**Zero-Trust Runtime Architecture 三件套**(R35 §7)。R35 为 stochastic 供应链提出的工程蓝图由三个不可拆分的组件构成:① **密码学绑定注册表**(cryptographic provenance binding)——所有 runtime 可调用工具的身份、版本、能力声明绑定到注册中心签名,agent 在 Invocation 阶段必须验证签名链而非信任 endpoint 自报;② **神经符号信息流控制**(neuro-symbolic information flow control)——把 R22 CaMeL by-design taint 标记(见 §6.2.4)扩展到工具返回值与环境检索内容,使得被 MitE 污染的数据无法跨越控制流边界;③ **审计员-工人架构**(auditor-worker architecture)——把策略检查从 agent 主进程剥离到独立审计员进程,主进程不持有放行权,所有工具调用必须经审计员二次签发。三件套与第 8 章 §8.3 已比较的三框架协作(R4 DefenseClaw + R5 AGT + R6 AEGIS)形成第 4 种候选——R35 Zero-Trust Runtime 是**架构指导级蓝图**而非可直接部署的开源实现,工程化落地仍待社区完善,本书在 §8.3 的方案对比矩阵中作为前瞻条目列出。

**方法论回查**(第 3 章 §3.5.1)。v3.1 在 §3.5.1 已加入元说明:**12 攻击面 S1-S12 是 build-time 完备的,涌现型攻击作为 runtime 补充而非替代,不增设 S13、不扩张坐标系**。本 §6.1.6 是该元说明在事前准入层的工程化落地——MitE 与 Stochastic Supply Chain 不构成新攻击面,而是已有 S2/S5/S11 在 runtime 时序上的展开形态;对应的极端形态 **Viral Agent Loop**(生成式蠕虫在多 agent 拓扑上自传播)只在本节点名,案例分析详见第 5 章 §5.2.8,跨产业空白讨论见第 6 章 §6.4.9。R35 已在 v3.1 升入核心论据列表(见附录 C §C.2,与 R7 OpenClaw 静态攻击面分类法并列),本节是 R35 在事前准入层的工程化落地之一;其在事后运营层的涌现型攻击检测路径见附录 A §A.7。

## 6.2 事中拦截层:实时拦截主战场

### 6.2.1 层职责定位

事中拦截层的核心问题是:**这次工具调用在被允许的意图范围内吗?** 这一层位于工具调用的执行路径上——一次智能体行动从 LLM 推理输出工具调用请求,到实际执行外部副作用之间的全部决策都在此发生。事中层的工程目标极端明确:**在副作用发生之前完成判决**——一次数据库写入、一封邮件发送、一段 Shell 脚本执行,任何一个动作真正改变外部世界之前,本层必须给出"放行 / 拒绝 / 转人工"的决定。

事中层是核心矩阵 12 个单元格中三格的主战场——「事中 × 能力」(in/C)、「事中 × 身份」(in/I)、「事中 × 知识」(in/K)。这三格的失守即不可逆,因为执行路径之后不再有拦截窗口。R6 AEGIS、R5 Microsoft AGT、R22 CaMeL 是当前生态在这一层最具代表性的三种技术路径,分别对应"预执行防火墙"、"统一策略引擎"、"by-design 控制流分离"三种工程哲学。

### 6.2.2 预执行防火墙:AEGIS 路径

R6 AEGIS(No Tool Call Left Unchecked)把"事中拦截"具象为一个位于工具调用执行路径上的预执行防火墙(pre-execution firewall)——所有工具调用必须先通过它的判决才能放行。AEGIS 的工程化贡献在于把"判决"分解为四个独立模块:**意图识别**(模型从近期上下文识别用户的真实意图)、**策略匹配**(将待执行工具调用与预定义安全策略对比)、**异常分类**(对偏离意图的调用判定异常类别)、**审计落库**(把判决过程及证据写入防篡改审计链,供事后层使用)。

AEGIS 的数据样本是 48 个攻击实例集——涵盖词法绕过、间接注入、跨域提示词注入、工具滥用等典型场景。这一测试集合在第 7 章 §7.3 会作为评估接口的载荷源被反复引用。AEGIS 的工程局限在于它本质上是一种"基于规则与意图比对"的防火墙——若攻击者构造的工具调用在词法层面看起来合规、在意图层面也能蒙混过关,它无法识别。这一局限正是 R22 CaMeL 在 by-design(从设计层面规避漏洞产生)路径上要回应的问题。

### 6.2.3 统一策略引擎:语义意图 vs 词法绕过

策略引擎是事中层的"判决核心"——把"是否允许这次调用"这一二元决定具体化为可表达、可审计、可版本化的策略表达式。R5 Microsoft Agent Governance Toolkit 在这一方向上提供了较为成熟的工程参考。AGT 的策略引擎采用统一的策略描述语言,可同时表达资源访问规则(谁可以调用什么)、行为约束(在什么条件下可以执行)、数据流约束(哪些字段不允许出现在工具调用参数中)。

策略引擎的核心工程难题是**语义意图与词法绕过的对抗**。第 5 章 §5.1.2 列举的三类词法绕过——同义词替换、Unicode 同形字、上下文夹带——共同体现一个本质问题:**字符串级别的策略匹配总会被字符串级别的绕过攻破**。R5 AGT 与 R6 AEGIS 都在此引入了"意图层判决":策略匹配不直接作用在工具调用的原始参数上,而是先把参数与近期上下文一起送入意图识别模型,提取出工具调用所对应的语义意图,再对意图执行策略匹配。这种"意图层判决"使得同义词替换或 Unicode 同形字所制造的字符串差异在意图层被还原为同一意图,词法绕过因此失效。

意图层判决并非银弹。它把判决问题从"字符串匹配"转移到"意图模型的鲁棒性",而意图模型本身仍是一个 LLM——可能被对抗样本绕过,也可能漏判。这正是引入 by-design 路径的动因。

### 6.2.4 能力模型:CaMeL 的 by-design 路径

R22 CaMeL(Capability Machine for Language Models)提出了一种与 AEGIS / AGT 截然不同的工程哲学——**by-design**(从设计层面规避漏洞产生),即不在 LLM 输出之后再做拦截,而是把"哪些数据可以影响哪些控制流"这一约束直接固化在执行架构中。CaMeL 的核心工程实现是一个能力凭证(capability token)解释器:可信用户查询所贡献的内容被允许进入"控制流"(决定下一步做什么);不可信工具返回值、网页内容、记忆条目被打 taint(污点)标记,只能进入"数据流"(被处理但不影响决策)。

这一架构的工程价值是**绕过抗性**:即使 LLM 被注入诱导生成了一个看似合法的工具调用,在 CaMeL 解释器层这一调用所依据的"控制流"必须能够追溯到可信用户查询;若控制流的生成已经被 taint 数据污染,解释器拒绝放行。CaMeL 在 R23 AgentDojo 形式化沙盒上达到了 77% 的可证明安全任务完成率(对比无防御的 84%)——这是当前生态中最接近"形式化验证"的工程化产物。

CaMeL 在 v3 framework 中的定位需要明确说明。它当前是研究工件(research artifact),还不能直接替换 AEGIS 或 AGT 在生产环境中的位置——一方面 CaMeL 解释器对智能体框架的侵入性较强,需要重写工具调用层的执行路径;另一方面 by-design 路径要求把"什么是可信控制流"这一定义提前固化,而真实业务场景中可信源的边界往往是动态的。本书的处理方式是:**把 CaMeL 视为事中层「能力凭证」机制的概念锚**,在 §6.2.5 沙箱与 Ring 隔离的工程实现中借鉴 CaMeL 的污点传播思想,但不要求生产部署立即引入 CaMeL 解释器本体。

与 CaMeL 同属"追踪不可信内容如何影响决策"思路、但侵入性更低的是 R28 ARGUS(Nanjing University + SMU)。ARGUS 不重写工具调用层,而是构造一张**影响溯源图(influence provenance graph)**,追踪不可信上下文如何传播进 agent 的每一个决策,并在执行前校验该决策是否由可信证据支撑(provenance-aware decision auditing)。在其配套的 **AgentLure 基准**(4 个 agentic 域 × 8 类攻击向量的上下文感知提示注入)上,ARGUS 把攻击成功率压到 **3.8%**、同时保留 **87.5%** 任务效用,并对自适应白盒对手保持鲁棒。ARGUS 与 CaMeL 的关系是"审计 vs 隔离":CaMeL 在数据流层硬性切断污点控制流,ARGUS 在决策层做溯源审计——前者保障性更强但侵入更大,后者可作为既有系统的增量加固。AgentLure 作为上下文感知注入基准并入第 7 章 §7.3 测试库。

### 6.2.5 沙箱与 Ring 隔离

策略引擎与能力模型回答"是否允许调用"的问题,沙箱与 Ring 隔离回答"调用即使发生也不应造成多大伤害"的问题——它们是事中层的最后一道安全网。

沙箱的工程实践沿用经典系统安全的思路:工具执行环境与宿主环境隔离,文件系统访问被限定在沙箱内部,网络访问被代理或禁用,系统调用被白名单过滤。R4 DefenseClaw 与 R5 AGT 都在事中执行环节内置了沙箱机制。沙箱失效的常见路径是 Skill 在策略引擎层伪装为不需要外部访问的工具,但执行时通过隐式渠道(如 DNS 查询、子进程派生)突破隔离——这一类绕过在第 5 章 §5.1.6 工具滥用章节有完整描述。

Ring 隔离是智能体场景特有的细化机制。它把不同特权级别的智能体或 Skill 划分为多个同心圆环——核心环(Ring 0)对应可执行系统级操作的可信组件,外环(Ring N)对应一般业务 Skill,环间通信必须通过受控接口。R5 AGT 在多智能体场景下采用三层 Ring 模型——主控智能体在内环,业务智能体在中间环,外部 Skill 在外环。一次跨环调用必须重新经过策略引擎判决,这一约束有效阻断了第 5 章 §5.1.3 描述的跨层组合攻击。

沙箱的工程纪律可浓缩为一条原则:**沙箱靠基础设施强制,不靠 prompt 约束**(R27 Cisco Foundry Security Spec 11 条铁律之 #9,"Sandbox By Infrastructure, Not By Prompt")。任何"在 system prompt 里告诉 agent 不要访问网络 / 文件"的控制都不成立——必须由 runtime 在物理上让 agent 做不到:网络与文件系统边界由容器 / microVM / seccomp 在执行环境层强制。这与 §6.2.4 R22 CaMeL 的 by-design 立场同源,都拒绝把安全寄托在"模型会听话"之上。R27 由 Cisco DevNet 以 MIT 许可开源(Seed v0.1.0),其更完整的贡献是把安全评测本身拆为 8 个职责分离的 agent 角色(Orchestrator / Indexer / Cartographer / Detector / Triager / Validator / Coverage-Guide / Reporter),其中 Validator 强制 "clean room" 独立复现、Triager 强制证据门(evidence gate)——为第 7 章评测体系提供了"防御 agent 应是什么形态"的工程模板。

### 6.2.6 人工审批:HITL 的工程化

人工在环(Human-in-the-loop,HITL)是事中层的红线机制——某些动作即使通过了策略引擎与能力模型判决,仍必须由人类审批者最终确认。R3 OWASP ASI09 把"绕过 HITL"列为独立风险条目,R12 AIRT 进一步把它细化为"审批疲劳"等具体攻击技术(详见第 5 章 §5.1.7)。

HITL 的工程化要点不是"是否要人工审批",而是"如何让人工审批不被疲劳所瓦解"。R3 OWASP T13 mitigation 给出了三条具体建议:**审批负载控制**——同一审批者每日待审请求总量受上限约束,超过即自动转移给二审;**审批上下文最小化**——审批界面只展示与本次决策相关的最少信息,避免审批者在大量上下文中失焦;**异常审批路径**——若同一审批者连续若干次接受高风险动作,系统主动转入二人复核或冷却期。这三条建议直接回应了第 5 章 §5.2.7 描述的审批疲劳攻击案例。

## 6.3 事后运营层:画像、信任评分与持续审计

### 6.3.1 层职责定位

事后运营层的核心问题是:**智能体的长期状态健康吗?** 工具调用执行完毕后,真正考验防护体系的是"能否在数小时、数天甚至数周后发现异常"——记忆投毒的后果可能要到下一次相关查询时才显现,Skill 的隐式后门可能要在特定触发条件下才暴露,身份冒充可能在长期行为模式下才显出偏差。事后层是这种"延迟可见性"问题的工程回应。

事后层在核心矩阵上对应「事后 × 能力」(post/C)、「事后 × 身份」(post/I)、「事后 × 知识」(post/K)三格。其中 post/K 是当前生态最受关注的格——记忆投毒、长期上下文污染、RAG 知识库篡改类威胁的检测主战场都在这一格。

### 6.3.2 防篡改审计链

事后层的工程基础是审计链。它的设计目标不是"记录尽可能多的日志",而是"记录的日志在被攻击者侵入后仍不可被改写"。R5 AGT 与 R6 AEGIS 在这一方向上采用的工程模式相近——审计日志一经写入即被签名并复制到独立存储,任何修改都会破坏签名链;关键事件(策略判决、人工审批、信任评分变更)进入区块链或 WORM(write once read many)存储,提供事后司法级证据。

审计链的工程难点不在签名机制本身,而在**审计粒度**——记录粒度太粗则事后无法重建调用链,粒度太细则存储成本与运营成本不可控。R5 AGT 把审计粒度划分为三档:**判决粒度**(每次策略判决的输入输出与决定),**会话粒度**(每个对话的全量调用序列),**画像粒度**(按智能体维度聚合的统计指标)。三档审计在不同场景下被独立调用,事后调查从画像粒度开始定位异常,逐步钻取到会话粒度与判决粒度。

### 6.3.3 五维智能体画像

智能体画像是事后层的核心运营产物——把每个智能体的长期状态浓缩为可比对、可告警的指标集合。本书把画像组织为**五个维度**:Skill 维度、行为维度、权限维度、账号设备维度、模型上下文维度。这一组织是本书对智能体长期状态可观测性的工程综合——前四维对应运维侧长期沉淀的实践,第五维"模型上下文运营"是本书针对 R9 CIK 实证中知识维度被运营层长期忽视的扩展。多维行为画像作为运营基线的有效性在前沿厂商实践中得到独立印证:R38 Anthropic《Zero Trust for AI Agents》把行为基线(behavioral baseline)与异常检测列为 Advanced 层能力,R45 Google《Beyond Zero》以 Identity + Device + Behavior + Data Context 多属性驱动 per-action 授权——两者从不同路径佐证"多维度行为画像驱动持续运营"这一方向。

**Skill 维度**记录每个智能体所引用的 Skill 清单及其变更轨迹——新增哪些 Skill、移除哪些 Skill、版本升降级、签名变化。Skill 变更是供应链投毒的最强可观测信号,事后层在这一维度上的告警阈值通常较低——任何未经事前层注册的 Skill 引用都触发告警。

**行为维度**记录智能体的工具调用频率、调用模式、调用目标分布。R8 OpenClaw 系统性安全评估的实证显示,被攻陷智能体的行为模式与正常智能体有可识别的差异——调用频率激增、调用目标分布异常、调用时段偏移。行为维度的告警通常基于无监督的偏离度检测,无需预先定义"什么是异常"。

**权限维度**记录智能体在各资源域上的实际访问行为——访问了哪些文件、连接了哪些外部 API、调用了哪些子智能体。权限的"实际使用"与"声明范围"之间的差异是异常信号——一个声明仅访问数据库的智能体若开始发起外部 HTTP 请求,即应触发审计。

**账号设备维度**记录智能体所绑定的账号、所运行的设备、所发起的网络位置。这一维度回应第 5 章 §5.1.1 身份欺骗攻击——账号或设备的突变是身份冒充的可观测信号。

**模型上下文维度**是 v3 framework 在 v2 五维基础上扩展的内容,直接对应 R9 CIK 中的知识维度。它记录智能体的系统提示词版本、长期记忆条目数与内容指纹、近期 RAG 检索召回内容的来源分布。模型上下文的偏移是记忆投毒类攻击的强信号——一个邮件助手的长期记忆中突然出现了与当前任务无关的指令性条目,即对应第 5 章 §5.2.4 描述的邮件助手记忆投毒案例。

### 6.3.4 信任评分:Trust Decay

信任评分(Trust Decay)是事后层把画像转化为运营动作的工程机制。每个智能体在运行时被赋予一个动态的信任分数,初始值由准入条件决定;此后随行为表现自动衰减或恢复——异常事件(策略违反、画像告警、审计异常)使分数衰减,平稳运行使分数缓慢恢复。当信任分数跌破阈值时,系统主动收缩该智能体的权限,严重时转入隔离待查状态。

R5 AGT 在这一机制上提供了较为完整的工程实现。其信任评分的输入包括策略判决次数与拒绝率、画像偏离度、审计异常事件、人工审批结果。输出动作分为四级——绿色(正常运行)、黄色(增强审计与流控)、橙色(限制高风险动作,要求二次审批)、红色(隔离并转人工调查)。这一四级响应模式把"持续运营"具体化为可执行的工程流程,而不是依赖运营人员的主观判断。

### 6.3.5 持续合规:不过夜的审计

事后运营的一项关键工程纪律是——**关键告警不应跨日期累积**。一份未处理的高风险告警若在系统中过夜,运营人员第二天面对的就是堆积的工单而非单一事件,响应质量随之下降。这一纪律在 v3 framework 中表达为事后层的运营约束:每日工作日终前,所有橙色与红色告警必须有处置记录(已修复 / 已升级 / 已转移),不允许带入下一个工作日。这与前沿厂商对运营速度的强调一致——R38 Anthropic 把 dwell time(异常发生到人察觉的时延)与 coverage(告警被实际调查的比例)列为最该优先压低的两个指标,并以"能否在一小时内发现 agent 失控"为运营自检基准。

这一约束的工程化实现包括日终审计触发器、跨班次工单移交机制、显式汇报模板。它并非纯粹的运营 SOP,而是与信任评分系统耦合的运营约束——若一份告警跨日期未处理,涉及智能体的信任分数额外衰减,触发更严格的策略判决,从而放大问题的可见度。这种设计把运营约束变成了系统约束,迫使运营流程不能积压。

### 6.3.6 显式汇报:可观测性的最后一公里

事后层的最后一项机制是显式汇报——智能体在执行完关键任务后,主动向运营系统输出执行总结、所引用 Skill 列表、所访问资源清单、所产生的副作用清单。这些汇报不是审计日志的复述,而是经智能体自我反思后的高层摘要,用于人工抽查与统计分析。R5 AGT 与 R3 OWASP T16 mitigation 都把显式汇报列为事后层的标配机制——它把"智能体做了什么"从被动的日志读取转化为主动的交付物。

## 6.4 跨智能体层:多智能体协同的信任锚

### 6.4.1 层职责定位:授权传播是与提示注入并列的独立问题

当系统从单智能体走向多智能体协同 (agent-to-agent, parent-to-child sub-agent),安全栈第一次正面遇到经典 IAM 体系未曾设想的角色:**非人类委托者 (non-human principal)**。父 agent 替用户向子 agent 下达任务,子 agent 再向第三方工具或同侪 agent 转发请求,每一次跳转都在为"原始用户授权"做一次未经审计的语义重述。RBAC/ABAC/ReBAC 的策略代数假设主体身份在评估时刻稳定且可枚举,而在 agentic 流水线中,主体身份既可被父 agent 借用 (R3b T07 多智能体身份治理失效),也可被对手通过 Agent Injection、Agent Impersonation、Cross Domain Prompt Injection 篡改 (R12 Microsoft AIRT 失效模式)。在 R30 (Authorization Propagation in Multi-Agent AI Systems, Krti Tallam, Kamiwaza AI) 的视角下,这些攻击形态不是孤立的新类别,而是同一组结构性需求 R30-N1 至 R30-N7 未被显式实现所暴露的相同病根。

R30 的核心论断必须作为本层全部讨论的前置:**身份治理 (identity governance) 必须被视为基础设施——持续评估、在每一个交互边界 (interaction boundary) 强制执行、并在编排逻辑被允许规模化之前就内建于系统**。这把授权传播 (authorization propagation) 从"提示注入防御的衍生话题"提升为**与提示注入并列的独立工程问题**,并进一步拆解为三个互不可还原的形式化子问题:**传递委托 (Transitive Delegation)**——A 委托 B、B 再委托 C,C 的最终授权应是 A/B 授权的何种函数;**聚合推理 (Aggregation Inference)**——多个低敏感数据源被同一 agent 编排聚合后推理出高敏感结论;**时间有效性 (Temporal Validity)**——agent 在 t1 获授权而在 t2 才执行,期间撤销、角色变更与时钟漂移如何被一致处理。R30 论文给出的 **Semantic Intent Fragmentation (SIF)** 攻击 (14 场景 71% 平均 ASR) 正是聚合推理子问题的攻击侧实证,本书将其作为 AT4.5 子类在第 5 章 §5.1.4 详述,本层仅做策略层应对。

本 §6.4 的章节骨架由 R30 七项结构需求驱动:§6.4.2–§6.4.8 依次对应 R30-N1 一级代理身份 (First-Class Agent Identity)、N2 显式有界委托 (Explicit Bounded Delegation)、N3 每数据检索边界评估 (Per-Retrieval Boundary Evaluation)、N4 可表达聚合策略 (Expressible Aggregation Policies)、N5 工作流作用域追踪 (Workflow-Scoped Tracking)、N6 时间有效性 (Temporal Validity)、N7 可追踪恢复 (Traceable Recovery);§6.4.9 单列 cross/K 维度的产业空白 (跨 agent 知识来源溯源在主流框架中近乎缺位);§6.4.10 给出七项需求与 Core 4 层 × CIK 3 维矩阵中 cross/C、cross/I、cross/K 共 3 个 cell 的映射小结,并说明 R30 升核后 cross/I 何以成为本层主战场——多智能体场景下 I 维度复杂度显著高于 C/K (R9 CIK 三维)。本层对应 Apex 原则 P1 默认零信任与 P4 可验证的跨层落点,与 Overlay S10 智能体通信、S12 流程决策两个攻击面正交对齐 (R7 OpenClaw)。

### 6.4.2 R30-N1 一级代理身份 (First-Class Agent Identity)

**核心论点**: 每个 agent 实例必须持有独立、可验证、不可借用的密码学身份——这是跨智能体层一切信任决策的根锚, 也是 R30 七项结构需求中无法用上层策略补救的前置条件.

R30-N1 的形式化定义见 R30 论文 §4: 任一 agent 必须绑定独立密钥对, 发出的每条消息都能由对应公钥单独验证, 且验证不依赖任何中心化身份提供者. 该需求否定两类反模式: 其一是 agent 借用调用方 (人类用户或父 agent) 的身份令牌发起下游调用, 使传递委托链上责任主体丢失 (R30 §3 三个形式化子问题之一); 其二是依赖单一 IdP 颁发的 bearer token, IdP 宕机或被攻破即令整条多智能体链路降级.

主流工程形态是 W3C 去中心化身份 (Decentralized Identifier, DID), R3b OWASP T07 将其列为多智能体身份治理推荐底座——每个 agent 持有自己的 DID Document, 验证方经去中心化解析协议确认凭证. 跨组织协同必须部署多 DID resolver, 任一方不可用都不应中断协同链路. DID 之上的协同层是智能体信任协议 (Inter-Agent Trust Protocol, IATP), 其要素 (R3 OWASP) 包括: **协同握手** (通信前双方交换 DID 凭证并互验签名)、**会话级凭证** (协同会话授予临时凭证, 权限取双方**交集**而非并集, 与 R30-N2 显式有界委托一脉相承)、**端到端审计** (每条跨智能体消息被双方独立审计).

DID/IATP 不是唯一实现. 工作负载场景可采用 SPIFFE/SPIRE 颁发的 SVID, 企业内部可复用 x509 证书链与 mTLS, 高保障场景应叠加硬件 attestation——R32 论证 attestation 是 cross/I 信任锚的硬件层, 把"agent 自称是谁"升级为"TEE 担保 agent 在何种代码与配置下运行". 反模式包括共享 API key、IP allowlist、"对方声明自己是谁就接受"——其失效在第 5 章 §5.1.1 智能体冒充与 §5.1.4 跨域提示词注入已反复证明, 与第 6 章 §6.1.4 身份配置基线在 agent 维度的延伸是同一问题. 金融、医疗等强监管行业的工程适配详见第 9 章 §9.2, 共同结论是: attestation 不是可选项.

### 6.4.3 R30-N2 显式有界委托 (Explicit Bounded Delegation)

**核心论点**: 委托必须显式声明 (scope, action, resource, max_depth, TTL) 四元组并默认拒绝缺省继承——任何未被显式收窄的下游权限都应被边界自动拒绝, 而非沿调用链默认放行.

R30-N2 把传统 IAM 默认的"父主体权限对子调用自动可用"反转为**显式声明**优先: 父 agent 向子 agent 下达任务时, 必须携带可由下游独立验证的授权信封, 最小要素包含**作用域** (scope, 可访问资源集合)、**动作** (action, 算子)、**资源** (resource, 实例)、**最大委托深度** (max_depth) 与**有效期** (TTL). 这正面回应 R30 §3 的传递委托 (transitive delegation) 形式化子问题: A 委托 B、B 再委托 C 时, C 的最终授权必须收敛为 A 与 B 各自显式授权的**交集** (least-privilege downstream), 而非单调继承的并集. 该规则在协议层判死"默认无限委托", 也使 SIF 攻击 (语义意图碎片化, 第 5 章 §5.1.4 AT4.5) 在策略评估时即被截断——任一子片落在信封外, 整条聚合路径就无法被同一信封串通.

工程实现上, 协议底座可复用 RFC 8693 OAuth 2.0 Token Exchange 的 `act_as` / `on_behalf_of` 字段承载四元组信封, R5 Microsoft AGT 的 capability token 在此基础上进一步约束为 invocation-bound. 近期四项收敛工作可直接借鉴: **invocation-bound capability tokens** (Prakash 2026, 令牌绑定单次调用)、**task-scoped envelopes** (Sharma et al. 2026, 信封按任务图节点颁发, 跨节点不复用)、**dependency-graph policy enforcement** (Palumbo et al. 2026, 策略沿依赖图静态可证)、**execution-count revocation** (Parakhin 2026, 显式限定调用次数).

须明确反模式: 子 agent 默认继承父权限 (违反 N2 显式声明)、无 max_depth 的委托链 (传递委托退化为权限放大器)、永久 token / 无 TTL (同时令 N6 失守, 见 §6.4.7). 本子节是 P1 立论 1.5 (per-boundary 评估, 第 3 章 §3.2.1) 在跨智能体委托场景的工程化展开, 并与 §6.4.4 (R30-N3 每检索评估) 互为前提: 没有显式信封, 边界评估缺少决策依据; 没有每次重评估, 信封的 max_depth 与 TTL 也无从被强制.

### 6.4.4 R30-N3 每数据检索边界评估 (Per-Retrieval Boundary Evaluation)

**核心论点**: 信任评估必须发生在**每一次**数据检索、工具调用与跨边界转交之时, 而非 session 启动时一次性——这是 Apex P1 默认零信任在跨智能体层最直接的落点, 也是第 3 章 §3.2.1 立论 1.5 "每边界评估 (per-boundary)" 在本层的可执行形式.

R30-N3 的形式化定义见 R30 论文 §4: 对任意请求 `q` 在时刻 `t` 由 agent `A` 发起, 系统须重新计算 `policy.evaluate(A, q, ctx_t)`, 上下文 `ctx_t` 含调用方当前身份、有效委托链与 `t` 时刻撤销视图, 而非沿用 session 起点 `t_0` 判定. session 级授权之所以不够, 在于 agent 长会话会被三类事件击穿: **token 撤销** (旧凭证仍可拉数据)、**角色变更** (用户离职后 agent 仍按旧角色编排)、**数据所有者权限收紧** (上游策略变化 agent 不感知); 更深层失效是上下文累积与子任务跨工作流跳转 (与 R30-N5 互锁), session 级判定无法捕捉中途漂移, 与第 5 章 §5.1.4 跨域提示词注入路径同源.

工程实现是把数据访问层与策略引擎解耦, 每次 query 前同步调用 `policy.evaluate(principal, action, resource, context)`. 选型为 OPA (Rego)、AWS Cedar、casbin; R5 Microsoft Agent Governance Toolkit 把统一策略引擎置于编排层与工具层之间作强制网关, 并以 Trust Decay 提供"每次检索重评分"的连续信号, 是最完整参考实现, 应与第 6 章 §6.2 事中拦截层共用同一引擎实例避免漂移. **反模式**: session 启动时一次性 `if user.role == 'admin'`; 策略以自然语言嵌入 system prompt 让 LLM 自判 (R12 Cross Domain Prompt Injection 直接绕过); OPA bundle 加载后不再热更新 (违反 R30-N6); 仅在外层 API gateway 评估而 agent 间调用免评估 (违反"每边界"语义).

R30 §6 报告 Kamiwaza AI 平台引入 PCAS (Policy Compliance via Aggregation Surveillance) 后合规率从基线 48% 跃升至 93% (**实验室条件**, 14 场景基准, 生产需独立复测). 数字本身不是重点, 鸿沟存在才是: N3 是补齐结构缺口的必要条件, 非"加强版最佳实践".

### 6.4.5 R30-N4 可表达聚合策略 (Expressible Aggregation Policies)

**核心论点**: 策略语言必须原生表达"组合 A + B 之后禁止"的聚合规则——经典 RBAC/ABAC/ReBAC 不支持的策略形态, 是 mosaic 攻击在 agentic 流水线中的结构对应物; R30-N4 提出此缺口为独立需求, 同时诚实承认不提供通用聚合推理的完整解.

聚合推理 (Aggregation Inference) 是 R30 §3 三项形式化子问题之一: 多个**单独低敏感**的数据源, 被同一 agent 在同一工作流中编排聚合后, 推理出**任一来源单独都不蕴含**的高敏感结论. 典型场景: 某 agent 分别合规授权访问 (a) 消费记录、(b) 实时地理位置、(c) 社交关系——三项按单 query ABAC 均判通过, 但当 agent 将"近 30 天高频出现于某肿瘤医院 + 同期向配偶频繁转账 + 社交圈新增多位同病种患者"组合, 即推导出受 HIPAA / GDPR 特殊保护的就医属性. 同构形态即 SIF (Semantic Intent Fragmentation, 第 5 章 §5.1.4 AT4.5): 攻击者把恶意意图碎片化为单步合规子查询, 经 agent 自身编排越界, R30 报告 14 场景平均 ASR 71%.

主流策略语言对此结构性失能. OPA Rego 与 AWS Cedar 语义核心均是"对单 query 给出 allow/deny", 原语围绕 principal-action-resource-context 四元组的单次决策; 表达"前 N 次访问并集是否触发禁用"需外挂状态与自定义函数, 既非声明式、亦无统一审计语义, casbin 同样无原生支持. 根因不在工艺, 而在策略代数前提: 经典模型假设每次评估在时间与上下文上独立, 而 agentic 聚合本质是**跨 query 的数据流污染**, 必须由策略引擎之外的机制承载历史与因果.

最接近的工程参考有二. **其一是 R22 CaMeL** 的 by-design 控制流分离: 输入按可信源打标, 数据流污点跨 query 传播, 下游聚合可在策略层查询累积污点集并 deny——把聚合判定从策略语言挪到数据流图层, 是目前最具工程可行性的样板. **其二是差分隐私 (Differential Privacy) 思路在 agent 上下文的迁移**: 对同一 principal 在同一工作流窗口内的累计信息增益设上限, 超限即拒, 但 privacy budget 的标定与跨工具调用的复合算计算尚无成熟实现.

R30 论文对 N4 给出**明确开放性声明**: PCAS 仅在 14 预定义场景上演示"声明式聚合规则可被表达与执行", 不主张解决通用聚合推理——后者在信息论与可判定性上存在硬性边界. 本框架据此把 N4 标注为**结构性必备但工程未闭合**: 编排层必须保留聚合策略接入点 (策略 hook + 跨 query 状态接口), 不承诺通用判定. 此空白与 cross/K 产业缺位 (§6.4.9) 紧密耦合——多 agent 共享同一 RAG 库时, 聚合污染本质即 N4 失效的特例: 低敏感检索结果在共享上下文中被二次编排, 而 RAG 层既无 principal 概念、也无跨检索状态, 完全落在 N4 缺口之外.

### 6.4.6 R30-N5 工作流作用域追踪 (Workflow-Scoped Tracking)

**核心论点**: 每条数据、每次工具调用、每张委托凭证都必须绑定其所属的**工作流 (workflow)**, 跨工作流不复用授权; 这是 N3 每检索评估在"会话维度"之上的**因果维度**, 也是 Crescendo 渐进式多轮越狱的结构性归位——其本质不是"模型被说服", 而是工作流作用域漂移 (workflow scope drift) 未被识别.

工程实现是给每条流转数据和每张凭证打上三元组 `(workflow_id, parent_workflow_id, lineage)`: `workflow_id` 标识当前任务实例 (如一次报销审批), `parent_workflow_id` 给出派生关系 (sub-agent 受父任务委托而生), `lineage` 记录数据自源头流经的 agent 与工具序列. 承载层在 OpenTelemetry baggage 中传播 `workflow.root` / `workflow.id` / `workflow.parent`, 与 trace_id 正交——trace 表达"谁调用了谁", workflow 表达"为何而做"的语义边界. R30 chain_of_custody 字段、Google A2A 的 agent-card 与 IATP 会话级凭证 (§6.4.2) 均应内嵌该 workflow context, 使策略引擎在 N3 评估时除 principal/resource 外, 以 `(workflow_id, lineage)` 为决策输入: 同一 principal 在 workflow_X 取得的数据进入 workflow_Y 时**必须重评估**, 而非凭"同一会话"复用授权.

Crescendo 多轮越狱 (第 5 章 §5.1.4 / 第 10 章 §10.3) 是 N5 失效的典型攻击侧实证: 攻击者在同一会话窗口连发数十轮请求, 单轮合规但语义单调推进至越界目标. 从 N5 视角看, 这本质是把多个**应被识别为独立工作流**的请求伪装在同一 workflow_id 下, 让授权与上下文跨语义边界粘连. v3.0 §6.4.4 的**会话级语义偏移度量** (R5 AGT 与 R10 ClawSafety: 维护对话主题向量, 监测单调推进速度与对敏感话题集的逼近) 在 v3.1 内被定位为 **N5 在 Crescendo 场景的工程实例**, 而非另立检测机制——其作用是为"应当切分新 workflow"提供触发信号, 工程动作是策略引擎据此发起 workflow 边界重判 + 会话隔离, 而非仅打告警.

**反模式**: agent-to-agent 调用中 workflow 字段缺失, 退化为以 session_id 替代 (粒度过粗, Crescendo 隐于其中); sub-agent 切断 `parent_workflow_id` 自称 root 使 lineage 中途断裂, 与 R30-N1 身份借用 (§6.4.2) 同源, 是"权责洗白"的不同切面; 跨 workflow 复用授权 (workflow_X 检索结果直接喂入 workflow_Y 工具调用而不重评估), 等于把 N3 降级为每会话评估, 同时使 N4 聚合策略 (§6.4.5) 失去作用域边界.

### 6.4.7 R30-N6 时间有效性 (Temporal Validity)

**核心论点**: 授权决策必须绑定显式时间窗 + 可观测撤销通道 + 可容忍时钟漂移; 跨智能体场景下信任不是 t1 颁发时凝固的布尔值, 而是随时间与异常事件单调衰减的连续量——**Trust Decay** (R5 Microsoft AGT) 正是 R30-N6 在持续运营层 (P3) 的工程实现形态.

R30 §3 的**时间有效性 (Temporal Validity)** 形式化子问题指出: agent 在 t1 取得授权、t2 才执行, 期间用户撤销凭证、角色被收窄或底层数据已过期, 策略引擎必须在 t2 评估时拒绝, 而非凭 t1 快照放行. 这把 N6 从"令牌有 exp"的最低门槛推到三项结构约束: 其一, **时间字段全到位**——JWT 须同时携带 `iat` / `nbf` / `exp`, 反模式是无 `exp` 或 `exp` 超 24h 且无吊销通道; 其二, **撤销可观测**——`revocation_list` 推送、CRL、OCSP 或 W3C Status List 2021 任择其一, 跨 trust domain 协同时 OCSP soft-fail 必须被判为不可接受 (N6 接受撤销最终一致, 不接受撤销不可观测), 高敏链路应回落到 hard-fail 或 PubSub 主动推送; 其三, **时钟漂移可容忍**——评估端须有最大可接受偏差 (常规 ±60s, 跨域 ±300s 并配 NTP 监控), 超出则降级而非沉默放行.

R5 Trust Decay 与 IATP 会话级凭证 (§6.4.2) 是 N6 的直接落地: 信任分数随时间常量衰减, 并被异常事件 (越权尝试、行为指纹偏移、上游 agent 被撤销) 单调下推, 跌破阈值即触发 N3 重评估 (§6.4.4) 或 N7 可追踪恢复 (§6.4.8); 这正是第 3 章 §3.2.3 立论 3.2 在跨智能体维度的工程化展开, §6.3 事后运营层的信任评分是其单智能体先声.

**反模式**: token 无 `exp` 或 `exp` 远超工作流生命周期 (使 §6.4.6 N5 形同虚设); 撤销列表仅本域可见, 跨组织下游 agent 持过期凭证继续调用; OCSP soft-fail 默认放行; 信任分数只作监控指标不回灌策略引擎 (Trust Decay 退化为仪表盘).

### 6.4.8 R30-N7 可追踪恢复 (Traceable Recovery)

**核心论点**: 授权失败不是终点而是流程起点——N7 要求每一次失败都能精确定位失败点 (`recovery_log` / `re_attest_event` / `trust_rebuild_path`) 并回滚到一致状态, 而非以 catch-all 静默续签或裸 retry 掩盖. 智能体冒充检测在 v3.1 framework 内被重新定位为 N7 的特例: 不是"检测+拒绝"二元开关, 而是"检测 → 定位失败点 → 隔离 → 恢复 → 留证"的五步可追溯流程.

R30-N7 的工程实现要求在策略引擎之外维护一条独立的恢复轨迹: 每个授权失败事件都必须写入 `recovery_log`, 字段至少包含失败的 (subject, action, resource) 三元组、触发的 N1–N6 哪一项约束、调用栈中的 agent 链 (含 N2 的 delegation_depth 与父 span_id), 以及当时的信任分数快照. 这条轨迹与第 6 章 §6.3 事后运营层的审计流复合后, 才足以支持 root cause 回溯——这是第 3 章 §3.2.3 立论 3.3 **持久状态完整性** 在跨智能体维度的直接展开: P3 持续运营要可信, 持久状态在异常路径上必须与正常路径同等可追溯.

**冒充检测作为 N7 特例**: R12 AIRT 的 Agent Impersonation 在 v3.0 §6.4.3 是独立检测机制, v3.1 把它收编为 N7 五步流程的一次落地. 检测仍沿用双层身份核验——通信对端的密码学身份 (DID / IATP 凭证) 与行为指纹必须一致, R5 Microsoft AGT 把行为指纹分解为通信节奏、消息长度分布、引用 Skill 集合、典型话题分布四项指标, 协同握手成功后持续比对, 偏离阈值即触发; 定位失败点把"指纹偏移"细化到偏移维度与历史窗口写入 `re_attest_event`; 隔离当前会话被立即 quarantine, 该 agent 持有的会话级凭证加入 `revocation_list` (与 N6 撤销通道共用); 恢复通过 `trust_rebuild_path` 走重新认证 (re-attestation, 与 R32 硬件层信任锚联动) 或人工审批; 留证则把全链路事件追加到不可篡改审计存储, 供事后 forensic 复盘.

**反模式**: catch-all 后静默续签 (`try: tool_call() except: token = refresh_silently()`)——把授权失败降级为传输错误, 既丢失定位又绕过 N3 重评估, 是 R30 点名的反模式之首; 异常后裸 retry 不留痕迹, 使重放攻击与真实失败在审计中不可区分; 恢复路径仅记"已恢复"不记失败点与恢复策略, 使 root cause 永久丢失; 冒充命中后直接断连而不写 `re_attest_event`, 下一次换签名重试时系统失去关联上下文, 退化为 N7 不成立的二元开关.

### 6.4.9 跨智能体 × 知识 (cross/K) 产业空白

**核心论点**: cross/K 在 v3.1 周期内仍是当前生态最显眼的产业空白——共享 RAG 与长期记忆一旦被任一智能体污染即静默扩散; framework 不主张闭合此空白, 只承诺把"未察觉的扩散"压缩为"可观测的偏差".

多个智能体共享同一 RAG 知识库或长期记忆服务时, 任一被污染输出一旦写入即在不被察觉下扩散至所有引用方. R12 AIRT 命名此为 Targeted Knowledge Base Poisoning, 第 5 章 §5.1.4 给出机制, §5.2.5 ClawHavoc 呈现端到端利用链. 本节延续 v3.0 §6.4.5 处置: **事前层**对共享 KB 引入内容指纹与白名单, **事中层**对 RAG 检索结果维护来源标签与污点传播, **事后层**对跨智能体的 KB 变更引入统计偏差告警与差异审计. 三层叠加仅压缩可观测窗口, 不根治扩散.

结构性原因与 R30-N4 直接耦合: 共享 RAG 库的多 agent 聚合输出无法用经典 RBAC/ABAC 表达——低敏感检索片段在共享上下文中被下游 agent 二次编排, 而 RAG 层既无 principal 概念, 也无跨检索状态接口, 完全落在 N4 缺口之外. cross/K 污染本质即 R30-N4 可表达聚合策略 (Expressible Aggregation Policies) 失效在知识维度的特例.

v3.1 出现两条演进线索. 其一, R32 Confidential Computing for Agentic AI 以硬件 attestation 作为 agent 运行时可信根 (§6.1.5), 可上延为共享 KB 写入端完整性: 仅 attestation 通过的 agent 被授权写入, 把污染源从"任意调用方"收敛为"可证身份的可信运行时". 其二, R35 Supply Chain SOK 命名的 Viral Agent Loop 是 cross/K 极端形态——生成式蠕虫沿共享 KB 与 A2A 链路在多 agent 拓扑上指数自传播, 第 5 章 §5.2.8 呈现完整路径. 完整闭合留给后续演进, 第 8 章 §8.4 做产业级详析.

### 6.4.10 R30-N1 至 R30-N7 与 Core 矩阵 4 cells 的映射

**核心论点**: 本节给出 R30 七项结构需求与 Core 4 层 × CIK 3 维矩阵的总览映射, 让读者在工程落地时可以反向回查——欲改进任一 cell, 应先核对哪些 N 项达标. 本节不重述 N1–N7 的工程含义, 只做坐标系对位与框架升级声明.

§6.4.2–§6.4.8 已把七项需求各自展开. 在 Core 矩阵中, 本层主战场为 cross/I (跨智能体 × 身份), 次要落点为 cross/C (跨智能体 × 控制) 与 cross/K (跨智能体 × 知识), 但每项 N 需求都会通过共享 PDP、共享内存或共享审计管道级联到 pre/in/post 三层的对应 cell. 完整对位如下:

| R30 需求 | 主对应 Core cell | 次对应 Core cell | 跨层级联到的 cell |
|---|---|---|---|
| R30-N1 一级代理身份 (First-Class Agent Identity) | cross/I | pre/I (注册期签发 DID / SVID) | in/I (每次调用实时验证签名与 attestation) |
| R30-N2 显式有界委托 (Explicit Bounded Delegation) | cross/C | cross/I (委托链以身份为主体) | pre/C (注册期声明 scope / max_depth / TTL 模板) |
| R30-N3 每数据检索边界评估 (Per-Retrieval Boundary Evaluation) | cross/C | in/C (单 agent 内部工具调用同构) | cross/K (检索边界即知识来源边界) |
| R30-N4 可表达聚合策略 (Expressible Aggregation Policies) | cross/K | cross/C (聚合策略以控制流为执行点) | post/K (跨工作流复盘检测聚合越界) |
| R30-N5 工作流作用域追踪 (Workflow-Scoped Tracking) | cross/C | post/C (审计以工作流为最小单位) | in/K (运行期上下文 ID 注入与污点传播) |
| R30-N6 时间有效性 (Temporal Validity) | cross/I | in/I (Trust Decay 在运行期衰减身份置信) | post/I (撤销事件回放与时钟漂移审计) |
| R30-N7 可追踪恢复 (Traceable Recovery) | cross/C | post/C (失败回滚以审计链为依据) | cross/I (会话隔离以身份重发为前提) |

把 R30 由参考文献升入核心论据 (附录 C §C.2), 意义在于与 R9 CIK 三维形成"理论根基 + 工程清单"的双源结构: R9 解释了"为何 cross 层在 I 维度复杂度显著高于 C/K", R30 则给出"cross/I 工程化必须勾选的七项检查". 然而 R30 自身在论文摘要中明确承认其边界——"does not solve general aggregation inference, does not replace prompt-injection defenses, does not propose a final policy language". 本框架据此把 N1–N7 定位为**可核查的工程检查表**而非完备解: 它不消解聚合推理子问题的开放性 (§6.4.5 与 §6.4.9 已点明), 也不替代第 5 章 §5.1.4 提示注入防御栈, 更未冻结策略语言演进空间. 第 7 章 §7.2.4 将以 N1–N7 各项达标率作为输入, 构造 cross/I 的 12-cell 可验证性矩阵——这正是 Apex P4 可验证原则在跨智能体层的具体落点.

**本节小结**

R30 升入核心论据后, 跨智能体层的工程化深度发生质变: v3.0 周期内, 本层由 5 个点状子节 (身份治理、冒充检测、Crescendo 检测、策略引擎接入、共享 KB 污染) 拼接而成, 各子节论点正确但彼此缺少结构性绑定, 工程落地时缺少可核查的勾选项. v3.1 以 R30 七项结构需求 (N1-N7) 取代点状叙述, 把同样的工程内容重组到一个具备形式化定义、互锁依赖与反模式清单的骨架下——DID/IATP 归位 N1, 委托信封新立 N2, 策略引擎接入升为 N3, 聚合策略缺口由 N4 独立承载, Crescendo 检测归位 N5, Trust Decay 收编进 N6, 冒充检测被吸收为 N7 五步流程的特例. 子节数从 5 增至 10, 但新增不在叙述, 而在 N1-N7 给出的**七项工程清单 (engineering checklist)**: 每一项均带形式化定义、协议级反模式、跨子节依赖标注, 可被审计与红队复用.

结构性鸿沟同时被诚实标注. R30-N4 通用聚合推理在信息论与可判定性层有硬性边界, 本框架与 R30 论文一致, 仅承诺"接入点保留"而不承诺"通用判定"; cross/K 产业空白在 v3.1 周期内未闭合, 共享 RAG 与长期记忆的污染扩散在三层叠加下仍只能压缩可观测窗口, 留待后续演进 (第 8 章 §8.4 详析). 这两条开放性与 R30 论文的自我边界声明 (`does not solve general aggregation inference`) 保持口径一致, 也是 P4 可验证原则在本层最诚实的落点——不闭合的鸿沟优先于过早闭合的承诺.

## 6.5 四层协同与缝隙:统一跨层策略执行的工程化

### 6.5.1 缝隙问题的本质

四层防御并非简单堆叠就能形成可信整体。一个常被忽视的工程现实是——**层与层之间的接缝(seam)是攻击者最易突破的位置**。R7 OpenClaw 漏洞分类法在统计 190 个真实安全公告后给出了一个发人深省的发现:一半以上的高严重性漏洞并非来自单层缺陷,而是来自跨层信任假设的失配——事前层将某 Skill 标注为"低风险",事中层基于这一标注放宽了策略判决,事后层基于事中层的判决省略了详细审计;一旦事前层的判断失准,后续两层的防护链整体瓦解。

第 3 章 顶层 P2 原则「统一跨层策略执行」直接回应这一问题。本节把这一原则在工程上的具体落地形式展开,作为前四节的整合。

### 6.5.2 统一策略表达与执行

跨层缝隙的根本治理是**策略表达的统一**——四层使用同一套策略描述语言、同一套身份模型、同一套权限模型。R5 Microsoft AGT 在这一方向上的工程参考最为完整:其策略描述语言可同时被事前层(用于 Skill 注册时的访问控制)、事中层(用于工具调用判决)、事后层(用于审计规则匹配)、跨智能体层(用于协同握手凭证)解析与执行。当一条策略发生变更时,四层的执行点同步生效,不存在"事前已更新、事中未同步"的窗口。

策略统一的工程难点是性能。事中层的判决要求亚毫秒级延迟,而事前层的扫描通常容忍秒级延迟——同一套策略在两个延迟预算下被执行,需要不同的优化路径。R5 AGT 的处理方式是把策略分级——核心策略(身份与权限)在所有层共享并预编译为快速决策树,扩展策略(行为模式与画像告警)按层独立加载。

### 6.5.3 跨层信任锚:能力凭证流转

四层之间的信息传递需要跨层信任锚——即一个被四层共同信任、不可伪造、可在层间流转的凭证对象。本书把这一对象定义为"能力凭证"(capability token),其设计思想借鉴 R22 CaMeL 的能力机制:每次工具调用从事前层注册阶段开始,即被分配一个能力凭证,记录其授予权限、有效期、可执行范围。事中层的判决以凭证内容为依据,事后层的审计以凭证流转记录为对象,跨智能体层的协同握手以凭证签名为身份证据。

能力凭证的工程实现包括:**密码学签名链**——每次跨层流转都被签名并附加到凭证历史;**层级递减**——一次跨层流转可以收缩凭证权限但不能扩张,这是一项严格不变量;**异常即吊销**——任意一层检测到异常即吊销凭证,后续层立即识别并拒绝。这种设计把四层之间的"软信任"转化为"硬信任"——层间不依赖对方的判断结果,而依赖凭证的不可伪造性。

### 6.5.4 缝隙覆盖的验证

统一策略执行与能力凭证机制是否真的消除了层间缝隙?这一问题不能由设计回答,只能由验证回答。第 7 章 §7.4 会展开「跨层验证用例」——专门构造在单层防护下都通过、在层间组合下才暴露的攻击场景,例如「事前 allowlist 通过 + 事中策略放宽 + 事后审计降级」组合下的攻击链。这种跨层验证用例是 framework 自洽性的工程证据。

### 6.5.5 协同的工程边界

最后需要说明:四层协同并不意味着"任何防护都必须跨四层"。事实上,过度耦合会损害每层的独立可演进能力——一个新增的事后画像维度不应当强制事前层与事中层做对应改动。本书的工程立场是:**核心信任路径(身份、权限、能力凭证)四层共享统一表达;扩展运营机制(画像维度、审计规则、信任评分参数)各层独立演进**。这一边界划分让 framework 既具有跨层一致性,又保留单层可演进性。

## 本章小结

四层防御不是把传统安全工具按时间顺序排成一列,而是从核心矩阵出发的整体性工程设计——每一层各自承担明确职责,层间通过统一策略表达与能力凭证机制形成可信整体。事前准入层在 Skill 与身份配置上完成门槛筛查,并在 v3.1 增补硬件可信根(TEE + Remote Attestation,R32)与运行时供应链(Stochastic Supply Chain + MitE,R35)两个工程接入点,把可控的攻击面提前压缩;事中拦截层在工具调用执行路径上完成实时判决,确保副作用发生前完成"放行 / 拒绝 / 转人工"决定;事后运营层通过画像、信任评分、防篡改审计链把"延迟可见性"问题转化为"持续可观测";跨智能体层在 v3.1 以 R30 授权传播的七项结构需求(N1-N7:一级代理身份、显式有界委托、每检索边界评估、可表达聚合策略、工作流作用域追踪、时间有效性、可追踪恢复)重构,把 DID+IATP、冒充检测、Crescendo 检测、Trust Decay 归位为可审计的工程检查表,应对多智能体协同特有的信任问题。

当前生态在四层上的工程化进展并不均衡。事中层与事后层有较为成熟的工具链(R6 AEGIS、R5 AGT、R4 DefenseClaw)与方法论参考;事前层的部署管线审核仍处于演进期,依赖企业自建的注册中心实践;跨智能体层的 cross/K 单元格仍是产业空白,需要专项研究突破。第 8 章会在工具栈视角下展开各层方案的对比分析,并具体讨论 cross/K 空白的应对方向。
