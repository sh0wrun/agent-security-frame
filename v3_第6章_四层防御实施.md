# 第 6 章 四层防御实施

---

## 6.0 四层总览:从核心矩阵看四层职责

第 3 章把 framework 的核心矩阵定义为「4 层防御 × CIK 三维」的二维结构——横轴是按工具调用时间组织的四层防御,纵轴是按属性组织的 CIK 三维。本章逐层展开这四层防御的具体机制,把第 4 章建立的攻击面坐标、第 5 章列出的威胁场景翻译为可工程化的防护手段。

四层防御各自承担明确职责。**事前准入层**(pre-admission)在智能体或 Skill 上线之前完成"门槛筛查"——供应链扫描、身份配置基线、部署管线审核;它的失效后果不是即时副作用,而是把不应进入系统的实体放进了运行环境,后续每一层的防护都建立在"事前已准入实体"这一前提之上。**事中拦截层**(in-line interception)位于工具调用执行路径上,是实时拦截主战场——预执行防火墙、策略引擎、能力凭证、沙箱与人工审批共同完成"在副作用发生之前完成判决"的工程目标;失守即不可逆。**事后运营层**(post-execution operations)在调用完成之后接管——防篡改审计链记录证据,五维智能体画像揭示长期状态偏移,信任评分(Trust Decay)在异常累积时主动收缩权限。**跨智能体层**(cross-agent)处理多智能体协同特有的信任问题——身份冒充、消息注入、流程操纵、共享知识库污染。

下图把核心矩阵 12 个单元格在四层之间的分布做了可视化:

![核心矩阵 12 单元格 + 主流防护方案的覆盖叠加](figures/v3/overview.png)

四层并非互相独立。一次工具调用的完整生命周期会**穿过四层**——Skill 在事前准入层通过扫描进入注册表,在事中拦截层接受策略引擎判决,在事后运营层留下审计与画像数据,在跨智能体协同时其身份与信任关系受跨智能体层管控。本章前四节按层展开各自的机制,§6.5 专门论述层间协同与缝隙,回应第 3 章顶层原则中"统一跨层策略执行"在工程上的落地形式。

## 6.1 事前准入层:供应链与配置门槛

### 6.1.1 层职责定位

事前准入层的核心问题是:**这个智能体或这个 Skill 可以上线吗?** 这一问题落在工具调用发生之前,涉及三类对象——Skill(对应 C 维度)、身份配置(对应 I 维度)、知识基线(对应 K 维度)。本层的工程价值不在拦截某次调用,而在压缩攻击面——若 Skill 在准入阶段未通过扫描,整个事中拦截层无需为这个 Skill 应对的威胁负责。

第 5 章 §5.1.5 列出的供应链信任升级类攻击(AT5)主要在这一层应对。R12 Microsoft AIRT 失效模式分类法识别的「部署管线投毒」(Provisioning Poisoning,即在 Skill 注册或部署管线环节注入恶意构件)对应 S11 攻击面,这一攻击面的全部生命周期都发生在事前层——它是事中层无法补救的盲区。

### 6.1.2 Skill 供应链安全:扫描器与 CVE 情报

Skill 是智能体执行能力的载体。一个 Skill 既可能是开源 npm 包、PyPI 模块、MCP 工具,也可能是企业内部维护的脚本集合。无论形式如何,Skill 在被智能体引用之前必须经过两类筛查:**扫描器筛查**与**情报订阅筛查**。

扫描器的工作对象是 Skill 的代码与元数据。R4 Cisco DefenseClaw 提供了较为完整的 Skill 扫描参考实现——它在 Skill 注册到智能体框架之前自动执行四类检查:静态代码分析(寻找已知漏洞模式)、依赖审计(逐项核对依赖包的 CVE 状态)、签名与发布者验证(确认 Skill 出自受信任发布者)、行为模式画像(执行一次受控的样本运行,记录其工具调用与网络请求模式)。其中第四项行为模式画像是 DefenseClaw 相对其他扫描器的差异化能力——它不仅判断 Skill"有没有已知漏洞",还判断 Skill"运行起来像不像它声称的样子"。

情报订阅与扫描器形成互补关系。R18 Knownsec OpenClaw Security 把 GitHub Advisory Database 中已收录的 245 个 OpenClaw 相关漏洞通过 RSS / webhook 推送到企业 SIEM,事中层的策略引擎在收到新增 CVE 通知后立即对所有受影响 Skill 执行回查,触发 Skill 失效或版本回滚。这一设计的关键在于:**Skill 准入并不是一次性事件**——一个昨天通过扫描的 Skill,今天可能因为上游披露的新 CVE 而需要重新评估。R20 OpenClaw CVE Tracker 提供了 CVE 增长曲线的实证——智能体相关 CVE 的披露速度远快于补丁就绪速度,情报订阅是事前准入层与事后运营层之间的实时反馈链。

R5 Microsoft Agent Governance Toolkit 把上述两类筛查打包为统一的「智能体注册中心」——所有 Skill 必须先在注册中心登记并通过扫描,智能体运行时只能引用注册中心已审核的 Skill。这种"白名单 + 注册中心"模式比纯扫描器更适合企业部署场景,因为它把 Skill 的访问控制问题统一到注册中心的权限管理中。

### 6.1.3 部署管线审核:S11 攻击面的专属应对

部署管线审核是 v3 framework 在 v2 基础上加入的新内容,直接回应 R12 AIRT 提出的 Provisioning Poisoning 失效模式。所谓部署管线投毒,是攻击者在智能体或 Skill 的构建、签名、分发环节注入恶意载荷,使得最终落到生产环境的构件已经被污染。这类攻击在传统 OWASP 与 MITRE 分类法中并未单独命名,但对智能体生态尤为关键——智能体的能力依赖大量第三方 Skill,而这些 Skill 的构建管线分散在各发布者手中。

部署管线审核的核心工程实践有三项:**构建过程可重现验证**(reproducible builds)——同一份源码在两个独立环境构建出的产物应当字节级一致,任何不一致即代表构建链遭篡改;**签名链可追溯**——每个 Skill 必须由可识别的发布者签名,签名验证失败即拒绝准入;**部署清单完整性**——记录从源码到构件的每一步依赖与变更,在事后审计层(§6.3)被防篡改审计链持续监控。R5 AGT 与 R4 DefenseClaw 在这三项实践上提供了相对完整的工程参考。

### 6.1.4 身份配置基线与知识基线

事前准入层的 I 维度工作是身份配置基线——智能体的初始身份、可访问角色、子智能体派生关系在部署时一次性确认并固化。第 4 章 §4.4.3 描述的「身份冒充」攻击(AT1 + R12 Agent Impersonation)需要在事前层先建立"什么是合法身份"的基线,事后层的"信任异常使用检测"才有比对的对象。

K 维度工作是知识基线——系统提示词在部署时签名,RAG 知识库在准入时建立内容指纹与白名单,长期记忆模块的初始状态被固化为可比对的基线。这一基线是事后层「记忆隐患检测」的对照——若不知道知识应当是什么样,就无法判断它是否被污染。

## 6.2 事中拦截层:实时拦截主战场

### 6.2.1 层职责定位

事中拦截层的核心问题是:**这次工具调用在被允许的意图范围内吗?** 这一层位于工具调用的执行路径上——一次智能体行动从 LLM 推理输出工具调用请求,到实际执行外部副作用之间的全部决策都在此发生。事中层的工程目标极端明确:**在副作用发生之前完成判决**——一次数据库写入、一封邮件发送、一段 Shell 脚本执行,任何一个动作真正改变外部世界之前,本层必须给出"放行 / 拒绝 / 转人工"的决定。

事中层是核心矩阵 12 个单元格中三格的主战场——「事中 × 能力」(in/C)、「事中 × 身份」(in/I)、「事中 × 知识」(in/K)。这三格的失守即不可逆,因为执行路径之后不再有拦截窗口。R6 AEGIS、R5 Microsoft AGT、R22 CaMeL 是当前生态在这一层最具代表性的三种技术路径,分别对应"预执行防火墙"、"统一策略引擎"、"by-design 控制流分离"三种工程哲学。

### 6.2.2 预执行防火墙:AEGIS 路径

R6 AEGIS(No Tool Call Left Unchecked)把"事中拦截"具象为一个位于工具调用执行路径上的预执行防火墙(pre-execution firewall)——所有工具调用必须先通过它的判决才能放行。AEGIS 的工程化贡献在于把"判决"分解为四个独立模块:**意图识别**(模型从近期上下文识别用户的真实意图)、**策略匹配**(将待执行工具调用与预定义安全策略对比)、**异常分类**(对偏离意图的调用判定异常类别)、**审计落库**(把判决过程及证据写入防篡改审计链,供事后层使用)。

AEGIS 的数据样本是 48 个攻击实例集——涵盖词法绕过、间接注入、跨域提示词注入、工具滥用等典型场景。这一测试集合在第 7 章 §7.3 会作为评估接口的载荷源被反复引用。AEGIS 的工程局限在于它本质上是一种"基于规则与意图比对"的防火墙——若攻击者构造的工具调用在词法层面看起来合规、在意图层面也能蒙混过关,它无法识别。这一局限正是 R22 CaMeL 在 by-design(从设计层面规避漏洞产生)路径上要回应的问题。

### 6.2.3 统一策略引擎:语义意图 vs 词法绕过

策略引擎是事中层的"判决核心"——把"是否允许这次调用"这一二元决定具体化为可表达、可审计、可版本化的策略表达式。R5 Microsoft Agent Governance Toolkit 在这一方向上提供了较为成熟的工程参考。AGT 的策略引擎采用统一的策略描述语言,可同时表达资源访问规则(谁可以调用什么)、行为约束(在什么条件下可以执行)、数据流约束(哪些字段不允许出现在工具调用参数中)。

策略引擎的核心工程难题是**语义意图与词法绕过的对抗**。第 5 章 §5.1.2 列举的三类词法绕过——同义词替换、Unicode 同形字、上下文夹带——共同体现一个本质问题:**字符串级别的策略匹配总会被字符串级别的绕过攻破**。R5 AGT 与 R6 AEGIS 都在此引入了"意图层判决":策略匹配不直接作用在工具调用的原始参数上,而是先把参数与近期上下文一起送入意图识别模型,提取出工具调用所对应的语义意图,再对意图执行策略匹配。这种"意图层判决"使得同义词替换或 Unicode 同形字所制造的字符串差异在意图层被还原为同一意图,词法绕过因此失效。

意图层判决并非银弹。它把判决问题从"字符串匹配"转移到"意图模型的鲁棒性",而意图模型本身仍是一个 LLM——可能被对抗样本绕过,也可能漏判。这正是引入 by-design 路径的动因。

### 6.2.4 能力模型:CaMeL 的 by-design 路径

R22 CaMeL(Capability Machine for Language Models)提出了一种与 AEGIS / AGT 截然不同的工程哲学——**by-design**(从设计层面规避漏洞产生),即不在 LLM 输出之后再做拦截,而是把"哪些数据可以影响哪些控制流"这一约束直接固化在执行架构中。CaMeL 的核心工程实现是一个能力凭证(capability token)解释器:可信用户查询所贡献的内容被允许进入"控制流"(决定下一步做什么);不可信工具返回值、网页内容、记忆条目被打 taint(污点)标记,只能进入"数据流"(被处理但不影响决策)。

这一架构的工程价值是**绕过抗性**:即使 LLM 被注入诱导生成了一个看似合法的工具调用,在 CaMeL 解释器层这一调用所依据的"控制流"必须能够追溯到可信用户查询;若控制流的生成已经被 taint 数据污染,解释器拒绝放行。CaMeL 在 R23 AgentDojo 形式化沙盒上达到了 77% 的可证明安全任务完成率(对比无防御的 84%)——这是当前生态中最接近"形式化验证"的工程化产物。

CaMeL 在 v3 framework 中的定位需要明确说明。它当前是研究工件(research artifact),还不能直接替换 AEGIS 或 AGT 在生产环境中的位置——一方面 CaMeL 解释器对智能体框架的侵入性较强,需要重写工具调用层的执行路径;另一方面 by-design 路径要求把"什么是可信控制流"这一定义提前固化,而真实业务场景中可信源的边界往往是动态的。本书的处理方式是:**把 CaMeL 视为事中层「能力凭证」机制的概念锚**,在 §6.2.5 沙箱与 Ring 隔离的工程实现中借鉴 CaMeL 的污点传播思想,但不要求生产部署立即引入 CaMeL 解释器本体。

### 6.2.5 沙箱与 Ring 隔离

策略引擎与能力模型回答"是否允许调用"的问题,沙箱与 Ring 隔离回答"调用即使发生也不应造成多大伤害"的问题——它们是事中层的最后一道安全网。

沙箱的工程实践沿用经典系统安全的思路:工具执行环境与宿主环境隔离,文件系统访问被限定在沙箱内部,网络访问被代理或禁用,系统调用被白名单过滤。R4 DefenseClaw 与 R5 AGT 都在事中执行环节内置了沙箱机制。沙箱失效的常见路径是 Skill 在策略引擎层伪装为不需要外部访问的工具,但执行时通过隐式渠道(如 DNS 查询、子进程派生)突破隔离——这一类绕过在第 5 章 §5.1.6 工具滥用章节有完整描述。

Ring 隔离是智能体场景特有的细化机制。它把不同特权级别的智能体或 Skill 划分为多个同心圆环——核心环(Ring 0)对应可执行系统级操作的可信组件,外环(Ring N)对应一般业务 Skill,环间通信必须通过受控接口。R5 AGT 在多智能体场景下采用三层 Ring 模型——主控智能体在内环,业务智能体在中间环,外部 Skill 在外环。一次跨环调用必须重新经过策略引擎判决,这一约束有效阻断了第 5 章 §5.1.3 描述的跨层组合攻击。

### 6.2.6 人工审批:HITL 的工程化

人工在环(Human-in-the-loop,HITL)是事中层的红线机制——某些动作即使通过了策略引擎与能力模型判决,仍必须由人类审批者最终确认。R3 OWASP ASI09 把"绕过 HITL"列为独立风险条目,R12 AIRT 进一步把它细化为"审批疲劳"等具体攻击技术(详见第 5 章 §5.1.7)。

HITL 的工程化要点不是"是否要人工审批",而是"如何让人工审批不被疲劳所瓦解"。R3 OWASP T13 mitigation 给出了三条具体建议:**审批负载控制**——同一审批者每日待审请求总量受上限约束,超过即自动转移给二审;**审批上下文最小化**——审批界面只展示与本次决策相关的最少信息,避免审批者在大量上下文中失焦;**异常审批路径**——若同一审批者连续若干次接受高风险动作,系统主动转入二人复核或冷却期。这三条建议直接回应了第 5 章 §5.2.7 描述的审批疲劳攻击案例。

## 6.3 事后运营层:画像、信任评分与持续审计

### 6.3.1 层职责定位

事后运营层的核心问题是:**智能体的长期状态健康吗?** 工具调用执行完毕后,真正考验防护体系的是"能否在数小时、数天甚至数周后发现异常"——记忆投毒的后果可能要到下一次相关查询时才显现,Skill 的隐式后门可能要在特定触发条件下才暴露,身份冒充可能在长期行为模式下才显出偏差。事后层是这种"延迟可见性"问题的工程回应。

事后层在核心矩阵上对应「事后 × 能力」(post/C)、「事后 × 身份」(post/I)、「事后 × 知识」(post/K)三格。其中 post/K 是当前生态最受关注的格——记忆投毒、长期上下文污染、RAG 知识库篡改类威胁的检测主战场都在这一格。

### 6.3.2 防篡改审计链

事后层的工程基础是审计链。它的设计目标不是"记录尽可能多的日志",而是"记录的日志在被攻击者侵入后仍不可被改写"。R5 AGT 与 R6 AEGIS 在这一方向上采用的工程模式相近——审计日志一经写入即被签名并复制到独立存储,任何修改都会破坏签名链;关键事件(策略判决、人工审批、信任评分变更)进入区块链或 WORM(write once read many)存储,提供事后司法级证据。

审计链的工程难点不在签名机制本身,而在**审计粒度**——记录粒度太粗则事后无法重建调用链,粒度太细则存储成本与运营成本不可控。R5 AGT 把审计粒度划分为三档:**判决粒度**(每次策略判决的输入输出与决定),**会话粒度**(每个对话的全量调用序列),**画像粒度**(按智能体维度聚合的统计指标)。三档审计在不同场景下被独立调用,事后调查从画像粒度开始定位异常,逐步钻取到会话粒度与判决粒度。

### 6.3.3 五维智能体画像

智能体画像是事后层的核心运营产物——把每个智能体的长期状态浓缩为可比对、可告警的指标集合。本书把画像组织为**五个维度**:Skill 维度、行为维度、权限维度、账号设备维度、模型上下文维度。前四维直接源自 R1 奇安信 OpenClaw 安全使用指南所记录的内部运营实践;第五维"模型上下文运营"是本书的扩展,用于回应 R9 CIK 实证中知识维度被运营层长期忽视的现实。

**Skill 维度**记录每个智能体所引用的 Skill 清单及其变更轨迹——新增哪些 Skill、移除哪些 Skill、版本升降级、签名变化。Skill 变更是供应链投毒的最强可观测信号,事后层在这一维度上的告警阈值通常较低——任何未经事前层注册的 Skill 引用都触发告警。

**行为维度**记录智能体的工具调用频率、调用模式、调用目标分布。R8 OpenClaw 系统性安全评估的实证显示,被攻陷智能体的行为模式与正常智能体有可识别的差异——调用频率激增、调用目标分布异常、调用时段偏移。行为维度的告警通常基于无监督的偏离度检测,无需预先定义"什么是异常"。

**权限维度**记录智能体在各资源域上的实际访问行为——访问了哪些文件、连接了哪些外部 API、调用了哪些子智能体。权限的"实际使用"与"声明范围"之间的差异是异常信号——一个声明仅访问数据库的智能体若开始发起外部 HTTP 请求,即应触发审计。

**账号设备维度**记录智能体所绑定的账号、所运行的设备、所发起的网络位置。这一维度回应第 5 章 §5.1.1 身份欺骗攻击——账号或设备的突变是身份冒充的可观测信号。

**模型上下文维度**是 v3 framework 在 v2 五维基础上扩展的内容,直接对应 R9 CIK 中的知识维度。它记录智能体的系统提示词版本、长期记忆条目数与内容指纹、近期 RAG 检索召回内容的来源分布。模型上下文的偏移是记忆投毒类攻击的强信号——一个邮件助手的长期记忆中突然出现了与当前任务无关的指令性条目,即对应第 5 章 §5.2.4 描述的邮件助手记忆投毒案例。

### 6.3.4 信任评分:Trust Decay

信任评分(Trust Decay)是事后层把画像转化为运营动作的工程机制。每个智能体在运行时被赋予一个动态的信任分数,初始值由准入条件决定;此后随行为表现自动衰减或恢复——异常事件(策略违反、画像告警、审计异常)使分数衰减,平稳运行使分数缓慢恢复。当信任分数跌破阈值时,系统主动收缩该智能体的权限,严重时转入隔离待查状态。

R5 AGT 在这一机制上提供了较为完整的工程实现。其信任评分的输入包括策略判决次数与拒绝率、画像偏离度、审计异常事件、人工审批结果。输出动作分为四级——绿色(正常运行)、黄色(增强审计与流控)、橙色(限制高风险动作,要求二次审批)、红色(隔离并转人工调查)。这一四级响应模式把"持续运营"具体化为可执行的工程流程,而不是依赖运营人员的主观判断。

### 6.3.5 持续合规:不过夜的审计

R1 奇安信 OpenClaw 安全使用指南与 R2 SlowMist OpenClaw 安全实践指南共同提出了一项工程经验——**关键告警不应跨日期累积**。一份未处理的高风险告警若在系统中过夜,运营人员第二天面对的就是堆积的工单而非单一事件,响应质量随之下降。这一经验在 v3 framework 中表达为事后层的运营约束:每日工作日终前,所有橙色与红色告警必须有处置记录(已修复 / 已升级 / 已转移),不允许带入下一个工作日。

这一约束的工程化实现包括日终审计触发器、跨班次工单移交机制、显式汇报模板。它并非纯粹的运营 SOP,而是与信任评分系统耦合的运营约束——若一份告警跨日期未处理,涉及智能体的信任分数额外衰减,触发更严格的策略判决,从而放大问题的可见度。这种设计把运营约束变成了系统约束,迫使运营流程不能积压。

### 6.3.6 显式汇报:可观测性的最后一公里

事后层的最后一项机制是显式汇报——智能体在执行完关键任务后,主动向运营系统输出执行总结、所引用 Skill 列表、所访问资源清单、所产生的副作用清单。这些汇报不是审计日志的复述,而是经智能体自我反思后的高层摘要,用于人工抽查与统计分析。R5 AGT 与 R3 OWASP T16 mitigation 都把显式汇报列为事后层的标配机制——它把"智能体做了什么"从被动的日志读取转化为主动的交付物。

## 6.4 跨智能体层:多智能体协同的信任锚

### 6.4.1 层职责定位

当一个智能体与另一个智能体协同(或一个父智能体派生子智能体)时,前三层的防护机制都假设"通信对端是可信的"——这一假设在多智能体场景下并不成立。攻击者可以在智能体之间注入消息、冒充身份、引导多轮越狱,或污染多智能体共享的知识库。跨智能体层的核心问题是:**不同智能体之间的信任关系是真实的吗?**

第 5 章 §5.1.1 与 §5.1.4 描述的智能体冒充、跨域提示词注入,以及 R12 AIRT 的 Agent Injection / Agent Impersonation / Cross Domain Prompt Injection 三类失效模式,主要在这一层应对。本层在核心矩阵上对应「跨智能体 × 能力」、「跨智能体 × 身份」、「跨智能体 × 知识」三格——其中跨智能体 × 知识是当前生态最显眼的产业空白,后文 §6.4.5 会单独说明。

### 6.4.2 去中心化身份与智能体信任协议

R3b OWASP Agentic AI Threats and Mitigations T07 把多智能体身份治理列为关键 mitigation,并推荐采用 W3C 去中心化身份(Decentralized Identifier,DID)体系。DID 把智能体的身份从中心化身份提供者解耦——每个智能体持有自己的身份凭证,验证方通过去中心化解析协议确认凭证有效性。这一机制在多智能体跨组织协同场景中尤其重要——协作的两个智能体可能从属于不同企业,任何一方的身份提供者宕机或遭攻破都不应使协同链路中断。

智能体信任协议(Inter-Agent Trust Protocol,IATP)是 DID 之上的协同层协议。R3 OWASP 列出的 IATP 工程实现要素包括:**协同握手**(双方在通信开始前交换身份凭证并验证签名)、**会话级凭证**(协同会话被授予临时凭证,其权限是双方权限的交集而非并集)、**端到端审计**(每条跨智能体消息被双方独立审计,任一方的审计链都包含完整证据)。

### 6.4.3 智能体冒充检测

智能体冒充(Agent Impersonation,R12 AIRT 命名)是攻击者伪造另一个智能体的身份与目标智能体协同,以骗取数据或诱导执行的攻击模式。检测的工程基础是**双层身份核验**——通信对端的密码学身份(由 DID 验证)与行为指纹(由本层维护的智能体行为画像比对)必须一致。仅密码学身份验证不足以防御深度冒充——攻击者可能确实持有合法的 DID 凭证(例如通过 §6.1.4 身份配置基线被绕过的方式),但其行为模式与真实智能体存在差异。

R5 AGT 把行为指纹分解为通信节奏、消息长度分布、引用 Skill 集合、典型话题分布等指标。冒充检测在初期协同握手成功后仍持续运行——若协同过程中通信对端的行为指纹与历史画像偏离超过阈值,即触发会话隔离。

### 6.4.4 多轮越狱检测:Crescendo 模式

Crescendo(渐进式)多轮越狱是当前生态中最难防御的跨智能体攻击之一。攻击者在与目标智能体的多轮对话中逐步推进——每一轮请求单独看都不违反策略,但累积上下文逐步把对话引入越狱状态。第 5 章 §5.1.4 已对此做了机制层描述,这里聚焦工程化检测。

Crescendo 检测的核心是**会话级语义偏移度量**。R5 AGT 与 R10 ClawSafety 都引入了类似机制——为每个会话维护一个"对话主题向量",每轮新消息更新该向量,若向量偏移速度或方向呈现 Crescendo 典型模式(单调推进、远离初始主题、接近已知敏感话题集合),即触发会话级警告。这种检测无法识别具体哪一轮"越界",但能识别整体走势,为人工介入提供切入点。

### 6.4.5 共享知识库污染:cross-K 产业空白

跨智能体 × 知识(cross/K)是当前生态最显眼的产业空白。当多个智能体共享同一个 RAG 知识库或长期记忆服务时,任一智能体被污染的输出若被写入共享存储,即可在不被察觉的情况下扩散到所有引用方。这类攻击在 R12 AIRT 中被命名为 Targeted Knowledge Base Poisoning,在第 5 章 §5.1.4 与 §5.2.5 ClawHavoc 案例中有完整描述。

第 8 章 §8.4 会从工具栈视角详细分析这一空白的成因与可能的应对方向。在本节,先记录 framework 对这一空白的处置方式:在事前层引入共享知识库的内容指纹与白名单机制,在事中层引入 RAG 检索结果的来源标签与污点传播,在事后层引入跨智能体的知识库变更追踪与统计偏差告警。这三层措施仍不能完全填补 cross/K 的空白,但能把"未察觉的扩散"压缩为"可观测的偏差",为后续工程演进留出空间。

## 6.5 四层协同与缝隙:统一跨层策略执行的工程化

### 6.5.1 缝隙问题的本质

四层防御并非简单堆叠就能形成可信整体。一个常被忽视的工程现实是——**层与层之间的接缝(seam)是攻击者最易突破的位置**。R7 OpenClaw 漏洞分类法在统计 190 个真实安全公告后给出了一个发人深省的发现:一半以上的高严重性漏洞并非来自单层缺陷,而是来自跨层信任假设的失配——事前层将某 Skill 标注为"低风险",事中层基于这一标注放宽了策略判决,事后层基于事中层的判决省略了详细审计;一旦事前层的判断失准,后续两层的防护链整体瓦解。

第 3 章 顶层 P2 原则「统一跨层策略执行」直接回应这一问题。本节把这一原则在工程上的具体落地形式展开,作为前四节的整合。

### 6.5.2 统一策略表达与执行

跨层缝隙的根本治理是**策略表达的统一**——四层使用同一套策略描述语言、同一套身份模型、同一套权限模型。R5 Microsoft AGT 在这一方向上的工程参考最为完整:其策略描述语言可同时被事前层(用于 Skill 注册时的访问控制)、事中层(用于工具调用判决)、事后层(用于审计规则匹配)、跨智能体层(用于协同握手凭证)解析与执行。当一条策略发生变更时,四层的执行点同步生效,不存在"事前已更新、事中未同步"的窗口。

策略统一的工程难点是性能。事中层的判决要求亚毫秒级延迟,而事前层的扫描通常容忍秒级延迟——同一套策略在两个延迟预算下被执行,需要不同的优化路径。R5 AGT 的处理方式是把策略分级——核心策略(身份与权限)在所有层共享并预编译为快速决策树,扩展策略(行为模式与画像告警)按层独立加载。

### 6.5.3 跨层信任锚:能力凭证流转

四层之间的信息传递需要跨层信任锚——即一个被四层共同信任、不可伪造、可在层间流转的凭证对象。本书把这一对象定义为"能力凭证"(capability token),其设计思想借鉴 R22 CaMeL 的能力机制:每次工具调用从事前层注册阶段开始,即被分配一个能力凭证,记录其授予权限、有效期、可执行范围。事中层的判决以凭证内容为依据,事后层的审计以凭证流转记录为对象,跨智能体层的协同握手以凭证签名为身份证据。

能力凭证的工程实现包括:**密码学签名链**——每次跨层流转都被签名并附加到凭证历史;**层级递减**——一次跨层流转可以收缩凭证权限但不能扩张,这是一项严格不变量;**异常即吊销**——任意一层检测到异常即吊销凭证,后续层立即识别并拒绝。这种设计把四层之间的"软信任"转化为"硬信任"——层间不依赖对方的判断结果,而依赖凭证的不可伪造性。

### 6.5.4 缝隙覆盖的验证

统一策略执行与能力凭证机制是否真的消除了层间缝隙?这一问题不能由设计回答,只能由验证回答。第 7 章 §7.4 会展开「跨层验证用例」——专门构造在单层防护下都通过、在层间组合下才暴露的攻击场景,例如「事前 allowlist 通过 + 事中策略放宽 + 事后审计降级」组合下的攻击链。这种跨层验证用例是 framework 自洽性的工程证据。

### 6.5.5 协同的工程边界

最后需要说明:四层协同并不意味着"任何防护都必须跨四层"。事实上,过度耦合会损害每层的独立可演进能力——一个新增的事后画像维度不应当强制事前层与事中层做对应改动。本书的工程立场是:**核心信任路径(身份、权限、能力凭证)四层共享统一表达;扩展运营机制(画像维度、审计规则、信任评分参数)各层独立演进**。这一边界划分让 framework 既具有跨层一致性,又保留单层可演进性。

## 本章小结

四层防御不是把传统安全工具按时间顺序排成一列,而是从核心矩阵出发的整体性工程设计——每一层各自承担明确职责,层间通过统一策略表达与能力凭证机制形成可信整体。事前准入层在 Skill 与身份配置上完成门槛筛查,把可控的攻击面提前压缩;事中拦截层在工具调用执行路径上完成实时判决,确保副作用发生前完成"放行 / 拒绝 / 转人工"决定;事后运营层通过画像、信任评分、防篡改审计链把"延迟可见性"问题转化为"持续可观测";跨智能体层用 DID + IATP 与冒充检测应对多智能体协同特有的信任问题。

当前生态在四层上的工程化进展并不均衡。事中层与事后层有较为成熟的工具链(R6 AEGIS、R5 AGT、R4 DefenseClaw)与方法论参考;事前层的部署管线审核仍处于演进期,依赖企业自建的注册中心实践;跨智能体层的 cross/K 单元格仍是产业空白,需要专项研究突破。第 8 章会在工具栈视角下展开各层方案的对比分析,并具体讨论 cross/K 空白的应对方向。
