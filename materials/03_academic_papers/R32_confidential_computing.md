## 一、元信息

- **作者**:Javad Forough、Marios Kogias、Hamed Haddadi(3 位)
- **机构**:**Imperial College London — Department of Computing**(单机构协作)
- **arXiv ID**:2605.03213(v2,2026-05-07 修订版)
- **学科**:cs.CR(交叉 cs.AI / cs.DC)
- **篇幅**:survey 形态(50+ 文献综述,具体页数待 PDF 核查)
- **核心问题**:Agent 在调用工具、执行用户委托、跨服务协同时不可避免地**处理大量敏感数据**(用户凭证、企业知识、API token、推理上下文);软件层防御(策略引擎、沙箱、prompt 过滤)在**敌对运维 / 特权对手(privileged adversary)** 威胁模型下根本不充分——root / hypervisor / cloud operator 可绕过所有应用层防御。**硬件 TEE(Trusted Execution Environment)如何在保留 agentic 自主能力的同时保护秘密?**
- **核心立论**(摘要演绎):**TEE / Confidential Computing 是 agentic AI 安全的硬件信任根**——remote attestation 提供"代码在可信硬件中运行且未被篡改"的密码学证明,enclave 内存隔离阻断特权对手对运行时秘密的访问。但 CC 是**必要但不充分**——它只解决"运行环境可信",不解决"语义完整性 / 内容级恶意输入"。

## 二、核心贡献

### 2.1 Agent-中心威胁模型:5 功能层 × 4 对手类

论文把 agentic AI 系统按**功能层(functional layer)**拆解为 5 层,把**对手(adversary)**按特权能力分 4 类,交叉得到 5 × 4 = 20 格威胁视图。这是当前 confidential computing 文献第一次以 **agent 为中心**(而非通用 ML pipeline 为中心)的威胁建模。

- **5 功能层**(具体层名待深读 §2 / §3 核实,推测含:用户交互层 / 推理层 / 工具调用层 / 记忆层 / 跨 agent 协同层)
- **4 对手类**(推测含:网络对手 / 受感染同机租户 / 特权运维 / 物理对手)

**工程意义**:这一矩阵恰好对仗本书 framework 的「4 层防御 × CIK 三维 = 12 单元格」核心矩阵——R32 是**威胁视角**的二维矩阵,framework 是**防御视角**的二维矩阵。两者在 v3.1 Ch6 §6.1 可以做对照引用。

### 2.2 TEE 平台对比:8 平台 × 9 安全目标

论文系统梳理当前主流 TEE / CC 平台并对 9 项安全目标做横向比较:

| 平台 | 类型 | 核心特性 |
|---|---|---|
| **Intel SGX** | CPU enclave | 细粒度页级隔离,enclave 内存上限制约大模型部署 |
| **Intel TDX** | VM-level CC | 全 VM 隔离,适合大模型 inference container |
| **AMD SEV-SNP** | VM-level CC | 内存加密 + nested page table 完整性 |
| **ARM TrustZone** | World 分区 | 移动端 / 边缘 agent 部署主流 |
| **ARM CCA** | Realm | TrustZone 的下一代,支持 confidential VM |
| **NVIDIA H100 CC** | GPU TEE | **首个生产级 GPU TEE**,支持机密 LLM inference |
| (其他 2 个) | (待 PDF 核实) | — |

**9 安全目标**(推测含:机密性 / 完整性 / 可验证性 / 新鲜性 / 抗物理攻击 / 抗侧信道 / 性能开销 / 大模型支持 / 多租户隔离 — 具体待深读)。

**关键性能数据**(_ANALYSIS 摘要):
- **Intel SGX enclave 内存限制 >30B 模型部署**——这是 SGX 路径在 agentic LLM 上的根本约束
- **NVIDIA H100 CC 模式 GPU overhead 4-8%**——可接受的性能成本,使生产部署成为可能
- **AMD SEV-SNP 内存隔离 209× CPU 周期开销**——内存访问微基准
- **CAEC 共享内存机制减 16.6-28.3% 足迹**——多 agent 共享 enclave 资源的优化

### 2.3 8 代表系统对标

论文挑选 8 个代表系统作为 confidential computing × agentic AI 当前 SOTA:

| 系统 | 定位 | 对 framework 的工具链候选意义 |
|---|---|---|
| **TEESlice** | TEE 内 ML 推理切分 | Ch8 §8.x 硬件层候选 |
| **CMIF** | confidential ML inference framework | Ch8 候选 |
| **TEECHAT** | TEE 内 LLM chat 系统 | Ch8 候选 + Ch6 §6.1 硬件可信根参考 |
| **Omega** | (推测为分布式 TEE 编排) | Ch6 §6.4 跨 agent attestation 候选 |
| **AttesMCP** | **MCP 协议的 attestation 扩展** | **Ch6 §6.4 + Ch8 MCP 工具链直接对接** |
| **CAEC** | Confidential Agent Execution Container | Ch6 §6.1 + Ch6 §6.2 沙箱升级 |
| **BlockA2A** | 区块链 + agent-to-agent attestation | Ch6 §6.4 跨 agent 信任锚 |
| (第 8 个) | (待 PDF 核实) | — |

**工程意义**:这 8 个系统正好填补 framework 当前 28 工具清单中**"硬件层"的空白**——v3.0 framework Ch8 工具栈聚焦扫描器 / 策略引擎 / 沙箱 / 审计层,完全没有 TEE / 硬件根信任类工具。

### 2.4 6 开放挑战

论文末段提出 6 项当前未解决的研究 / 工程挑战,直接构成 framework 未来演进的潜在方向:

1. **多跳 attestation**(multi-hop attestation):agent A 调 agent B 调 agent C 时,如何把 attestation 信任链传递不丢失?
2. **TEE-backed RAG**:RAG 知识库与检索过程如何在 enclave 内运行,既保护用户查询又保护知识库 IP?
3. **CC 感知协议**(CC-aware protocols):MCP / A2A / IATP 等协议如何原生集成 attestation?
4. **GPU-TEE 性能优化**:H100 CC 4-8% overhead 仍有压缩空间,LoRA 微调 / KV cache 加密等新场景待解决
5. **DOR 合规**(Data Operating Residency / Data-at-Rest 合规):TEE 在 GDPR / CCPA / 国内数据出境合规中的法律地位
6. **模型供应链完整性**:从训练 → 微调 → 部署的全链路 attestation(与 R35 Supply Chain SOK 有强重叠)

### 2.5 关键工件(论文产出的可复用构件)

- **5 维分析框架**:功能层 × 威胁 × 安全目标 × CC 机制 × 部署形态 — 后续研究可直接套用
- **Agent-CC 信任边界双层模型**:agent 内信任边界(enclave 内 / 外)+ agent 间信任边界(attestation 协议层)
- **Threat-to-Defense 矩阵**:把 20 格威胁视图直接映射到 CC 机制 — 这是 framework 第 4 章攻击面建模可借鉴的呈现形态

### 2.6 论文自述的局限

- **CC 不解决语义完整性**:enclave 保护"运行环境",不保护"输入内容"——prompt injection / 间接注入照样有效
- **多代理认证与新鲜性未成熟**:跨 agent attestation 链 + replay 防御仍是开放问题
- **侧信道泄漏持续存在**:cache timing / power analysis 等 side-channel 攻击 TEE 至今未完全免疫

## 三、与 v3 framework 的对接点

R32 在 framework 上的"防护形状" = **以 P4 可验证为主轴的硬件根信任补充**。R32 不替换任何现有章节内容,而是**填补 framework 在敌对运维 / 特权对手威胁模型下的盲区**。五处对接:

### 3.1 第 6 章 §6.1 事前准入层 — 新增 §6.1.5 硬件可信根(主对接点)

**v3.0 §6.1 当前内容**:Skill 供应链扫描器(§6.1.2)/ 部署管线审核(§6.1.3)/ 身份配置基线 + 知识基线(§6.1.4)。整节聚焦**软件供应链 + 配置基线**,**没有任何硬件信任根章节**——这是一个 framework 自承的盲区:若运维 root 或云厂商被攻破,§6.1.2 的扫描结论与 §6.1.3 的签名链都失去信任锚。

**v3.1 §6.1 重写建议**:在 §6.1.4 之后新增 **§6.1.5 硬件可信根:TEE 与 Remote Attestation**,作为 framework 在敌对运维威胁模型下的兜底层。建议结构:

```
§6.1.5 硬件可信根:TEE 与 Remote Attestation(v3.1 新增)
  §6.1.5.1 威胁模型扩展:从"被入侵的同租户"到"敌对运维"(R32 4 对手类引入)
  §6.1.5.2 TEE 平台选型矩阵:SGX / TDX / SEV-SNP / TrustZone / CCA / H100 CC
       — 关键决策点:enclave 模型大小约束 + GPU 性能预算 + 多租户场景
  §6.1.5.3 Remote Attestation 作为 P4 可验证原则的硬件实现
       — attestation 报告进入 §6.5.3 能力凭证链作为初始信任锚
  §6.1.5.4 与 §6.1.3 部署管线签名链的衔接:attestation 证明"签名链在可信硬件上执行"
  §6.1.5.5 局限说明:CC 不解决语义完整性,必须与事中层 §6.2.2-6.2.4 软件防御组合
```

**与 §6.1 既有内容的衔接**:R32 的硬件信任根不是替代 §6.1.2 Skill 扫描或 §6.1.3 签名链,而是把它们**锚定到不可被特权对手篡改的根**——即"我相信 Skill 扫描结果"的前提是"我相信扫描器本身在 TEE 中执行且未被改写"。

### 3.2 第 6 章 §6.4 跨智能体层 — 新增 §6.4.6 attestation-based 协同信任锚

**v3.0 §6.4 当前内容**:DID + IATP(§6.4.2)/ 智能体冒充检测(§6.4.3)/ Crescendo 多轮越狱(§6.4.4)/ cross-K 产业空白(§6.4.5)。当前 §6.4 的信任锚**纯软件**——DID 签名 + 行为指纹 + 会话级语义偏移,全部假设"通信对端的运行环境本身可信"。

**v3.1 §6.4 重写建议**:在 §6.4.5 之后新增 **§6.4.6 attestation-based 协同信任锚**,引用 R32 的 AttesMCP / BlockA2A / Omega 三个代表系统作为工程参考。建议要点:

- **多跳 attestation 链**:agent A → B → C 协同时,每跳交换 attestation report,组合为信任链(对应 R32 §2.4 开放挑战 1)
- **CC 感知协议(CC-aware MCP)**:AttesMCP 是 MCP 协议层引入 attestation 的工程参考,与 v3.1 inbox 的 MCP 簇形成自然对接
- **跨 agent attestation 在 cross/I 单元格的定位**:R32 attestation = cross/I 维度的**身份可验证性证明**,与 R5 AGT DID / IATP(软件身份)、R30 N1 一级代理身份(身份不变量)形成"硬件 + 协议 + 不变量"的三层身份治理叠加

**与 §6.4.5 cross-K 产业空白的关系**:R32 §2.4 开放挑战 2(TEE-backed RAG)直接指向 cross-K 空白的潜在硬件解法——若共享 RAG 知识库在 enclave 内运行,污染源可被 attestation 链定位。但论文承认这是开放问题,不是已有方案。本节应明示"硬件方向是 cross-K 空白的研究路径之一,工程落地仍待成熟"。

### 3.3 第 3 章 §3.2.4 P4 可验证 — 加硬件实现注解

**v3.0 §3.2.4 当前内容**:P4 原则定义"可验证的安全保障"——独立验证系统持续运行,声称安全 vs 证明安全的差距。当前论证主要靠**软件验证**(红蓝对抗 / 形式化基准 / 审计链)。

**v3.1 §3.2.4 加注建议**:在 P4 原则段末加一段约 150 字的注解:

> **P4 的硬件实现层**:可验证的安全保障在最严格的威胁模型(敌对运维 / 特权对手)下,需要硬件根信任作为软件验证的最终锚点。Remote Attestation(R32)是 P4 在硬件层的工程实现——它把"验证者相信执行环境未被篡改"的信任问题归约到"验证者相信硬件厂商根证书"。在不涉及云厂商或本地运维潜在威胁的部署场景中,P4 的软件实现(R6 AEGIS 审计链 / R23 AgentDojo 形式化基准)足够;在金融 / 医疗 / 政府等高敏感场景下,P4 应叠加 R32 硬件根信任。详见 Ch6 §6.1.5。

### 3.4 第 8 章 工具链选型 — 新增"硬件层"工具类别

**v3.0 Ch8 当前内容**:28 工具按"扫描器 / 策略引擎 / 沙箱 / 审计 / 测试集合 / 智能体平台"等软件类别组织,**完全没有 TEE / 硬件根信任工具**。

**v3.1 Ch8 加节建议**:新增一个工具子类别 **"硬件可信根与机密计算"**,纳入 R32 的 8 代表系统作为候选:

- **TEECHAT**:TEE 内 LLM chat 部署参考(对应高敏感场景客服 / 医疗咨询 agent)
- **CAEC**:Confidential Agent Execution Container(对应需要多 agent 共享 enclave 的金融场景)
- **AttesMCP**:MCP 协议层 attestation 扩展(对应需要可验证 MCP 工具调用的合规场景)
- **BlockA2A**:区块链 + A2A attestation(对应跨企业 agent 协同场景)
- **NVIDIA H100 CC**:GPU TEE 生产基础设施(对应所有需要机密 LLM inference 的场景)

**与 v3.0 Ch8 的关系**:硬件层工具**不替换**扫描器 / 策略引擎,而是与它们叠加 — 一个完整高敏感部署 = 软件防御层(R6 AEGIS + R5 AGT)+ 硬件信任根(R32 系列)+ by-design 数据流隔离(R22 CaMeL)。

### 3.5 第 9 章 §9.2 行业适配 — 金融 / 医疗 / 政府场景的 TEE 必要性说明

**v3.0 §9.2 当前内容**:按金融 / 医疗 / 政府 / 制造等行业讨论合规与适配。当前章节强调**合规要求**(GDPR / HIPAA / 国内数据合规)但**未提硬件层强化**。

**v3.1 §9.2 加段建议**:在每个高敏感行业子节末加一段"硬件可信根强化建议",引用 R32:

- **§9.2.x 金融场景**:支付 / 风控 agent 处理 PCI-DSS 数据,推荐 NVIDIA H100 CC + AMD SEV-SNP(对应 R32 §2.4 开放挑战 5 DOR 合规)
- **§9.2.x 医疗场景**:涉及 PHI 的 agent 推理与 RAG 推荐 TEECHAT 类方案(enclave 内运行 LLM)
- **§9.2.x 政府场景**:涉密 agent 部署强制要求 TEE,attestation 报告作为合规审计证据

## 四、与已知 baseline 的关系

R32 与 framework 内已有 R 编号形成**"软件信任 + 硬件信任 + by-design + 内容拦截"四角互补**关系:

| 关系类型 | 对标 R 编号 | 对比要点 |
|---|---|---|
| **互补 — 软件信任根 vs 硬件信任根** | R5 Microsoft AGT(DID / Ed25519 / IATP) | **R5 是软件信任根**(密码学身份 + 策略协议),**R32 是硬件信任根**(enclave + attestation)。R5 假设"运行环境可信"才有效;R32 提供"运行环境本身可被密码学证明"。**两者叠加 = 完整身份治理栈** |
| **互补 — 内容拦截 vs 数据保护** | R6 AEGIS | **R6 拦截恶意内容**(预执行防火墙判决工具调用),**R32 保护敏感数据**(enclave 内存不被特权对手读取)。R6 解"输入语义", R32 解"运行环境"。**两者关注完全不同的维度,无竞争关系** |
| **互补 — by-design 数据流 vs 硬件数据隔离** | R22 CaMeL | **R22 by-design 数据流隔离**(taint 控制流 vs 数据流),**R32 硬件数据隔离**(enclave 内存加密)。**两者叠加 = 软硬协同的数据隔离**——R22 在应用层阻断 taint 数据影响控制流,R32 在硬件层阻断特权对手访问任何层数据 |
| **互补 — 身份不变量 vs 身份硬件根** | R30 Authorization Propagation(若升核心) | **R30 N1 一级代理身份是不变量层**,**R32 attestation 是身份的硬件实现层**。R30 回答"身份应满足什么属性",R32 回答"身份属性如何被密码学锚定到硬件" |
| **强重叠 — 供应链方向** | R35 Supply Chain SOK(若升核心) | **R35 的运行时供应链 zero-trust runtime architecture** 与 **R32 §2.4 开放挑战 6 模型供应链完整性** 高度重叠 — 都要求从训练 → 部署全链路 attestation。**v3.1 章节稿应在 §6.1.5 + R35 的"Stochastic Supply Chain"子节做交叉引用** |
| **威胁视角对仗** | framework Ch3 12 单元格核心矩阵 | R32 的 5 功能层 × 4 对手类 = 20 格威胁视图,与 framework 4 防御层 × CIK 三维 = 12 单元格在维度上对仗。**两者可在 Ch6 §6.1 引言段做"威胁 vs 防御"二维对照** |

**"软 + 硬 + by-design + 内容" 四角关系图**(R5 / R32 / R22 / R6 在 framework 中的定位):

```
                软件信任根              硬件信任根
              (R5 AGT / DID)         (R32 TEE / attestation)
                   |                          |
                   +----- 身份治理栈 ----------+
                                |
              by-design 数据流          内容拦截防火墙
              (R22 CaMeL)              (R6 AEGIS)
                   |                          |
                   +----- 控制/数据流栈 -------+
```

## 八、不应该做的事

- ❌ **不要把 R32 写成 framework 的"硬件层 = 第 5 防御层"** — framework 的 4 层防御(事前 / 事中 / 事后 / 跨智能体)是按**工具调用时间**组织的,不按"软件 vs 硬件"组织。硬件可信根是 §6.1 事前准入层的**实现层补充**,不是与 4 层并列的新层。增设"第 5 层"会破坏核心矩阵的二维结构,违反 framework 的层次稳定性
- ❌ **不要把 TEE / Remote Attestation 当作"解决所有 agent 安全问题的银弹"** — R32 论文自承 CC 不解决语义完整性 / 内容级恶意输入 / 侧信道泄漏。在 §6.1.5 章节稿中必须显式写"TEE 是必要但不充分,必须与 §6.2 事中拦截层、§6.3 事后运营层、§6.4 跨智能体层组合使用"。否则会误导读者把硬件信任根当成 prompt injection 的解药
- ❌ **不要在 §6.4.6 章节稿中把 BlockA2A 的"区块链 + agent attestation"作为通用推荐方案** — BlockA2A 是论文列举的 8 系统之一,代表一种研究路径,**不是生产成熟方案**。当前生态对"区块链 + AI agent"的工程价值仍有争议(性能开销 / 监管不确定性 / 运维复杂度)。引用时应标"研究参考",不应放在与 AttesMCP / CAEC 同等的工程推荐位
- ❌ **不要让 §6.1.5 硬件可信根章节产生"必须强制采用 TEE"的合规暗示** — framework 的定位是"工程化指南 + 行业适配建议",不是合规强制要求。TEE 在非高敏感场景下的性能 / 成本开销可能不合理。§6.1.5 应明确写"硬件可信根是高敏感场景的可选强化路径,而非所有 agent 部署的默认要求",留出"软件防御足够"的合理路径
- ❌ **不要忽视 R32 与 R35 在"模型供应链完整性"上的重叠** — R32 §2.4 开放挑战 6 与 R35 Supply Chain SOK 的运行时供应链有强重叠。v3.1 章节稿在 §6.1.5(R32)与 R35 对应章节之间应有**显式交叉引用**——否则两个章节会重复表达,读者认知负担加大。建议 §6.1.5.4 加一句"R35 的运行时供应链 zero-trust runtime architecture 在应用层给出本节硬件根信任的应用对接方案"
