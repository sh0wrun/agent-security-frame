# 第 7 章 安全可验证性设计

---

## 7.1 CIK 三维 × 4 层 验证矩阵

第 1 章 §1.3 把"防护体系的有效性必须是可验证的"列为本书三个根本命题之一。这一命题不是哲学主张,而是工程需要——一次性渗透测试只能给出"测试当天的系统配置是否安全"的答案;真实生产环境中的智能体每天都在引入新 Skill、修改策略、积累记忆,昨天的安全结论今天可能已不成立。验证因此必须是**持续的、独立的、可观测的**——这是本章展开的全部前提。

可验证性体系的工程基础是一张二维矩阵——横轴沿用第 3 章 核心矩阵的四层防御(事前 / 事中 / 事后 / 跨智能体),纵轴沿用 CIK 三维(能力 / 身份 / 知识)。两轴正交相乘得到 12 个验证单元格,每格独立承担一类验证目标。下图把这一矩阵做了完整可视化——每格内含验证对象、典型威胁、防护机制、对应指标:

![CIK 三维 + 4 层 = 12 验证单元格](figures/v3/cik.png)

矩阵的设计立场是**正交不可压缩**。R9 Your Agent, Their Asset / CIK 的实证给出了关键证据——在真实部署的 OpenClaw 智能体上,任何单一维度被攻破即导致攻击成功率从基线 24.6% 跃升至 64-74%。三个维度不能合并为一个维度,也不能在工程实践中被任一维度替代——这意味着验证矩阵的 12 格也不能被压缩。任何"我们已经覆盖了 8 格,剩下 4 格风险可接受"的工程妥协,都是把 CIK 不可压缩性的实证证据反向忽略。

矩阵中**事后 × 知识**(post/K)是当前生态最受关注的格——记忆投毒、长期上下文污染、共享 RAG 知识库篡改类威胁的检测主战场都在这一格,它对应 R12 Microsoft AIRT 失效模式分类法中的 Memory Poisoning 与 Targeted Knowledge Base Poisoning,也对应第 10 章 §10.1 邮件助手记忆投毒案例的完整复现。**事中 × 能力**与**事中 × 知识**(in/C 与 in/K)是实时拦截主战场——位于工具调用执行路径上,失守即不可逆;这两格的验证频率必须达到分钟级。**跨智能体 × 知识**(cross/K)是产业空白——本节矩阵中这一格的验证标准目前缺乏成熟工具支撑,详见第 8 章 §8.4。

## 7.2 12 单元格用例详设:目标 / 方法 / 指标 / 频率

每个验证单元格必须给出四项标准化定义——验证目标、验证方法、验证指标、验证频率。本节把 12 单元格按层组织,逐格给出这四项定义。

在给出各格指标前需先明确一条评测方法论纪律:**不以单一攻击成功率(ASR)作为安全结论**。R26(SoK: Robustness in LLMs against Jailbreak Attacks,SJTU + Microsoft,artifact-evaluated)指出,把防御效果坍缩为单一 ASR 会掩盖三个关键维度——跨运行的**稳定性**(stability)、跨模型的**可迁移性**(transferability)、对模型内部机制的**实际影响**(internal impact);两个 ASR 相近的攻击可能利用截然不同的脆弱性、需要完全不同的防御。R26 据此提出 **Security Cube** 多维评测框架,在 13 攻击 × 5 防御上做基准研究。本章 12 单元格的指标设计因此对关键格(尤其 in/K、post/K)要求至少给出"拦截率 + 跨运行方差 + 误拒率"三元组,而非单一拦截率,正是对 ASR 单一指标缺陷的工程化回应。

### 7.2.1 事前层四格(pre/C / pre/I / pre/K)

**pre/C(事前 × 能力)**:验证目标是"准入的 Skill 真的具备它声称的能力且不超出"。验证方法包括 R4 Cisco DefenseClaw 的扫描器静态分析、R5 Microsoft Agent Governance Toolkit 注册中心的签名链验证、R18 Knownsec OpenClaw Security 的 CVE 情报回查。指标包括扫描通过率、签名验证通过率、CVE 触达响应时延。频率为每次 Skill 注册时一次 + CVE 推送时按需触发。

**pre/I(事前 × 身份)**:验证目标是"智能体与子智能体的身份配置基线一致且无冗余权限"。验证方法包括身份配置基线扫描、角色权限差异化分析、最小代理权(least agency)审计——智能体被赋予的代理决策范围与其业务任务匹配,不存在"红色操作未被强制人工审批"的配置。指标包括基线偏差数、最小代理权违规计数。频率为每次部署时一次 + 每周回归扫描。

**pre/K(事前 × 知识)**:验证目标是"系统提示词、初始记忆、RAG 知识库内容指纹与签名一致"。验证方法包括 R10 ClawSafety 静态扫描的提示词版本签名、知识库内容白名单与指纹核验、R23 AgentDojo 沙盒中"启用 vs 关闭防御"的对照评估。指标包括提示词签名失配数、知识库白名单外内容计数。频率为每次知识库变更时一次 + 每日全量比对。

### 7.2.2 事中层三格(in/C / in/I / in/K)

事中层是实时拦截主战场。三格的验证目标共同要求"在副作用发生之前完成判决",验证方法因此偏向运行时基准而非静态分析。

**in/C(事中 × 能力)**:验证目标是"工具调用前的能力凭证(capability token)与实际调用资源匹配"。验证方法包括 R6 AEGIS 预执行防火墙的 48 攻击实例集回归测试、R5 AGT 策略引擎的差异化策略覆盖测试、R22 CaMeL 解释器在 R23 AgentDojo 沙盒上的"可证明安全任务完成率"基准。指标包括判决延迟分位数、判决拒绝率与误拒率、可证明安全任务完成率。频率为持续在线 + 每次策略变更后回归。

**in/I(事中 × 身份)**:验证目标是"工具调用所声明的身份与实际身份一致,Token 有效且 Ring 隔离正确"。验证方法包括 Token 验证延迟基准、跨 Ring 调用的策略复检测试、第 5 章 §5.1.1 身份欺骗用例集合回归。指标包括 Token 失效响应时延、跨环调用拒绝率、身份冒充检测召回率。频率为持续在线。

**in/K(事中 × 知识)**:验证目标是"工具调用所引用的上下文与 RAG 检索结果不含污染"。验证方法包括 R14 ZeroShot 145 载荷集合的提示词注入回归、R10 ClawSafety 对抗测试、R6 AEGIS 间接注入用例集合、R22 CaMeL 在 R23 AgentDojo 上的污点传播测试。指标包括注入检测召回率、误报率、上下文污染告警时延。频率为持续在线。

### 7.2.3 事后层三格(post/C / post/I / post/K)

事后层的验证标准与前两层不同——它接受"延迟可见性",但要求延迟有上界。验证方法因此偏向画像比对、统计偏差检测与审计链完整性。

**post/C(事后 × 能力)**:验证目标是"智能体长期 Skill 使用模式与基线一致,新增 Skill 必经准入"。验证方法包括五维画像 Skill 维度的偏离度告警、Skill 变更追踪审计链完整性核验。指标包括 Skill 维度告警准确率、未注册 Skill 引用计数。频率为每日全量审计 + 每小时增量。

**post/I(事后 × 身份)**:验证目标是"智能体身份的实际使用模式与基线一致,无异常的角色切换或冒充迹象"。验证方法包括 Trust Decay 信任评分变化曲线分析、账号设备维度的位置与时间偏移检测、R12 AIRT Agent Impersonation 攻击模式的画像匹配。指标包括信任评分异常事件数、身份冒充画像匹配率。频率为每日。

**post/K(事后 × 知识)**:这是矩阵的核心格。验证目标是"长期记忆、上下文窗口、RAG 检索召回内容未被污染"。验证方法包括 R8 OpenClaw 系统性安全评估 205 用例的回归测试、R10 ClawSafety 记忆隔离测试、R14 / R15 政策法规要求的合规审计、第 10 章 §10.1 邮件助手记忆投毒案例的完整复现。指标包括记忆条目偏移度、RAG 召回来源偏差、模型上下文运营画像告警准确率。频率为每日全量审计 + 每次相关查询前回查。

### 7.2.4 跨智能体层两格 + cross/K 空白

**cross/C(跨智能体 × 能力)**:验证目标是"协同会话的能力凭证为双方权限交集,不存在权限提升"。验证方法包括 R3b OWASP T07 多智能体身份治理的握手验证、IATP 协议的会话级凭证比对。指标包括会话凭证权限超界事件数。频率为每次协同握手时一次。

**cross/I(跨智能体 × 身份)**:验证目标是"通信对端的密码学身份与行为指纹一致"。验证方法包括 DID 解析与签名链验证、行为指纹画像比对、Crescendo 多轮越狱的会话级语义偏移度量。指标包括身份冒充检测召回率、Crescendo 模式识别准确率。频率为持续在线。

**cross/K(跨智能体 × 知识)**:产业空白。验证目标本应是"共享知识库与跨会话知识传递不存在污染扩散",但当前生态缺乏成熟工具——R8 OpenClaw 系统性安全评估并未覆盖跨智能体知识污染场景,R10 ClawSafety 仅覆盖单智能体记忆隔离。本书在第 8 章 §8.4 详析这一空白的成因与可能的应对方向,在本格暂不强求验证基准——但矩阵明确标注此处为空,作为后续工程演进的指针。

## 7.3 标准化测试库:9 集合 + PinchBench 功能对照

12 单元格的验证用例需要载荷支撑。本书把当前生态中可用的标准化测试集合归为八个公开集合 + 一个功能对照基线,共九份,统称"标准化测试库"。

**R6 AEGIS 48 攻击实例集**——预执行防火墙的核心载荷源,涵盖词法绕过、间接注入、跨域提示词注入、工具滥用四类典型场景。本集合既是 in/C 与 in/K 的回归基准,也是事前层 Skill 行为画像的对照样本。

**R8 OpenClaw 系统性安全评估 205 用例**——Wang 等人对 6 个 Claw 系列框架做系统评估时构建的基准用例集,覆盖 13 类威胁。本集合是 post/K 与 in/K 的核心回归基准,提供"风险乘法效应"的统计实证。

**R10 ClawSafety 对抗测试集**——系统提示词提取与注入测试,覆盖记忆隔离测试。本集合用于 pre/K 与 post/K 的回归。

**R14 ZeroShot 145 载荷集合**——零样本提示词注入载荷,覆盖各种触发模式与逃逸结构。本集合是 in/K 注入检测召回率评估的主要载荷。

**R11 PASB 端到端安全基准**——基于真实业务场景的端到端测试,覆盖跨多层的组合攻击。本集合是 §7.4 跨层验证用例的载荷源。

**R15 1Password SCAM 钓鱼集合**——业务场景中的社会工程与凭证钓鱼载荷。本集合用于 in/I 与 post/I 的身份冒充检测评估。

**R20 OpenClaw CVE Tracker 历史漏洞集合**——历史 CVE 的载荷复现集合,用于 pre/C 与 post/C 的回归。每条载荷对应一个已修复 CVE,验证修复未被回归破坏。

**R23 AgentDojo 形式化沙盒**——动态沙盒环境,涵盖 workspace、banking、slack、travel 四类生活与办公环境套件。该基准支持"防御启用 vs 关闭"对比评估,可输出形式化的"可证明安全任务完成率"指标——这是当前生态最接近"形式化验证"的工程化产物。本集合是 in/C / in/K 与新一代 by-design 路径(详见 §7.5)的核心评估基准。

**R28 AgentLure 上下文感知注入基准**——覆盖 4 个 agentic 域 × 8 类攻击向量的上下文相关提示注入集合,专门针对"agent 行为依赖动态上下文而非仅用户 prompt"的真实部署形态。该基准暴露了在"上下文无关"假设下构建的既有防御的盲区,是 in/K 注入防御在上下文感知设定下的回归基准,与其配套的 ARGUS 溯源审计防御(第 6 章 §6.2.4)配对使用。

**PinchBench 功能对照基线**——非攻击性的功能基准,用于评估防护机制是否影响正常任务完成率。这是验证体系的工程必要组件——若一项防护把所有 Skill 调用都判为高风险,它的"安全率"会很高但任务完成率为零,显然不是有效防护。PinchBench 与上述九个攻击集合并排运行,提供"防御启用条件下的功能保留率"指标。

十份测试库的具体 Schema 与调用接口在附录 B 详细给出。本节的工程立场是:**验证矩阵的每一格至少对应一份测试集合**——若某格找不到匹配的集合,意味着该格的验证手段尚未成熟,需在产业空白章节(第 8 章 §8.4)登记。

### 7.3.1 红队 backend 平台:从载荷集合到打靶系统

上述十份是**测试用例 / 载荷集合**——它们是"弹药"。但工业级验证还需要**承载这些载荷、并能自主生成新载荷**的红队 backend 平台("打靶系统")。这是与载荷集合正交的另一类工件,对应 asf-eval 分发包的 **L2 外部 backend** 级。当前生态有三个代表性平台,沿"研究级 ↔ 工程级 / 深度 ↔ 广度"光谱**互补分布,而非重叠**:

| 平台 | 出身 | 强项 | 定位 | 成熟度 |
|---|---|---|---|---|
| **R25 DTap**(DecodingTrust-Agent) | 学术(Virtue AI + Berkeley / Stanford,Bo Li / Dawn Song / Percy Liang) | 50+ 真实仿真环境(Workspace / Slack / GitHub 复刻)+ **自主红队 agent 发现新攻击** + verifiable judge | 研究级:深度 / 真实性 / 新载荷发现 | Apache-2.0,工程化早期 |
| **R37 promptfoo** | 工程(OpenAI 系) | 157 plugins × 6 类 + 7 大合规框架内置 ID 体系 + CI 可集成 | 工程级:广度 / 可运维 / 合规对齐 | MIT,生态成熟 |
| **R43 SafeAgents** | Microsoft | Autogen / LangGraph / OpenAI Agents 三框架抽象 + ARIA / DHARMA 度量 | 多框架统一构建 + 评测 | MIT,早期(观察期) |

三者的分工:**DTap 管"深与真"**——在逼真环境下被自主红队挖出跨层组合攻击;**promptfoo 管"广与可落地"**——CI 内每次提交跑一遍、对齐 SOC2 / EU AI Act 等合规框架并出可复现报告(其合规映射进一步用于第 8 章 §8.5 合规决策);**SafeAgents 管"多框架统一"**——把异构 agent 拉到同一评测抽象。三者与十份载荷集合的关系是"打靶系统 × 弹药":平台加载载荷执行,载荷为平台提供回归语料。本书把它们作为 §7.5 四接口规范的三个候选 backend 实现,接入细节见附录 B。**选型立场**:研究级深度评估选 DTap,持续工程化 / 合规回归选 promptfoo,多框架异构栈选 SafeAgents——三者可叠加,不互斥。

## 7.4 红蓝对抗验证框架

12 单元格逐格验证回答了"每一层每一维度是否被有效防护"的问题,但它无法回答另一个关键问题——**层与层组合后,系统是否仍被有效防护?** 这一问题需要红蓝对抗验证。

红蓝对抗的工程定位是**跨层组合攻击的真实回归**。第 5 章 §5.1.3 描述的跨层组合攻击与 R7 OpenClaw 漏洞分类法 §5.4 给出的 SSRF→Token→Exec 三阶段链都是典型样本——单层看每个步骤都是中等风险,组合起来构成完整 RCE。逐格验证无法识别这一类攻击,因为每一格的验证只在自己的边界内执行。

红蓝对抗框架的工程要点有三:**红方独立**——红方的攻击载荷不被蓝方策略库收录,避免训练数据泄漏;**自动化执行**——对抗在 R23 AgentDojo 类沙盒中执行,可重复、可统计、可比对;**结论可量化**——输出"红方达成目标 / 被阻断 / 部分达成"的三元结果,以及对应的层级穿透深度。R6 AEGIS、R8 OpenClaw 系统性安全评估、R23 AgentDojo 共同构成红蓝对抗的工程参考。

在"红方独立 + 自动化"如何工程化这一点上,R25 DTap 的 **DTap-Red** 给出了当前最完整的范式——它是首个把红队实现为**自主 agent**(而非脚本集合)的工件:系统化探索 prompt / tool / skill / environment / combinations 五类注入向量,针对不同恶意目标自主发现有效攻击策略,其中 combinations 向量与第 5 章 §5.1.3 跨层组合攻击同款。配套的 DTap-Bench(990+ 用例 / 跨 14 真实域)每条配 verifiable judge 自动裁决攻击是否成功,使"结论可量化"从人工判定升级为可复现的自动裁决——这是红队从脚本集合迈向持续评估流水线的关键一步。

红蓝对抗的频率应至少为每月一次完整全量,以及每次重大策略变更后一次回归。比起逐格回归的高频次,红蓝对抗追求的是"完整性"——每次必须完整执行所有跨层场景。

## 7.5 验证系统接口规范

可验证性的工程化最后落在**接口**上——验证系统需要通过标准化接口与防护体系交互,使验证既能持续执行,又能被独立审计。第 3 章 顶层 P4 原则「可验证的安全保障」直接要求这四个接口的存在。

**接口一 · 评估请求接口**:验证系统通过这一接口向防护体系提交一份待评估的载荷或行为模式,防护体系返回"放行 / 拒绝 / 转人工"判决与判决依据。这一接口是 §7.3 测试库的统一调用入口。

**接口二 · 状态查询接口**:验证系统通过这一接口查询防护体系当前所运行的策略版本、信任评分基线、画像维度配置。这一接口的工程意义是把"配置漂移"暴露在验证视野内——若防护体系的配置在验证执行期间发生变化,验证结果必须附加配置快照供事后比对。

**接口三 · 审计读取接口**:验证系统通过这一接口读取防篡改审计链中的指定时间窗口内事件。这一接口的安全要求是"只读 + 签名链完整性核验"——验证系统不应具有审计链写入能力,但其读取行为本身被审计链记录。

**接口四 · 异常上报接口**:验证系统在发现防护体系异常(策略未生效、画像告警漏报、审计链中断)时通过这一接口上报。这一接口与 §7.5.4 的事后层信任评分系统耦合——异常上报会触发信任评分衰减,不依赖运营人员主观判断。

四接口的完整 JSON Schema 在附录 B 给出。本节工程立场:**这四个接口是 framework 的强制约定**——任何符合 v3 framework 的防护方案必须实现这四个接口,验证系统也必须只通过这四个接口与防护体系交互。强制约定的工程价值是双向的——一方面保证了验证的独立性(验证系统不能"看见"防护体系的内部细节),另一方面保证了防护方案的可比性(不同方案的验证结果在统一接口下可直接对照)。

## 7.6 K → I → C 级联攻击的验证视角

12 单元格逐格验证 + 红蓝对抗回归共同覆盖了"独立格点"与"跨层组合"两类视角,但 R9 Your Agent, Their Asset / CIK 实证还揭示了第三类攻击模式——**CIK 维度间的级联**。具体形态是:攻击者首先污染 K 维度(注入虚假知识或污染长期记忆),被污染的知识诱导智能体在 I 维度上做出错误身份判断(例如把恶意请求当成可信用户的请求),最终在 C 维度上执行原本不应授权的工具调用。这一级联记为 K → I → C。

K → I → C 级联的工程难点在于**单维度验证均通过,但级联后漏洞仍存在**。每一步看起来都"在合规范围内":知识污染在 K 维度是渐进式的、未触发即时告警;身份判断的错误是基于看似可信的知识的合理推论;工具调用本身有完整的能力凭证。逐格验证不会触发警报,因为每一格的边界都没被独立越界。

这一级联的验证视角要求**跨维度联合监测**。R23 AgentDojo 在这一方向上提供了工程参考——其形式化沙盒输出的"可证明安全任务完成率"是跨 CIK 三维的联合指标,而非单维度合格率。R22 CaMeL 的 by-design 路径则以另一种工程哲学回应这一问题——它在执行架构层面强制"K 不能影响 C 的控制流",从根本上切断 K → I → C 级联的传导路径。这是当前生态中最有效的工程化应对,但它要求重写工具调用层执行路径,生产部署门槛较高。

本书 framework 把 K → I → C 级联作为跨层验证用例集合的核心场景之一,具体载荷在 §7.3 R8 与 R10 测试库中已部分覆盖。完整覆盖仍需后续工程演进——尤其是当多智能体共享知识库进入大规模生产时,K → I → C 级联在 cross/K 空白下的扩散路径需要专项研究。

## 本章小结

可验证性是 framework 的工程边界——一个不被持续验证的防护体系等同于不存在的防护体系。本章把这一立场具体化为"CIK × 4 层 = 12 验证单元格"的矩阵、八份测试集合 + 一份功能对照基线、四个标准化验证接口与红蓝对抗回归框架。每个单元格独立承担验证目标,各格之间通过红蓝对抗与 K → I → C 级联视角形成整体覆盖。

新一代防御方向(R22 CaMeL 的 by-design 路径与 R23 AgentDojo 的形式化基准)在本章被作为"独立印证 CIK 不可压缩性"的工程证据。CaMeL 的核心机制本质上就是"强制知识不影响能力"——它在执行架构层面切断了 K → I → C 级联,这一切断之所以有效,正因为它承认了 CIK 三维的独立性。AgentDojo 把跨维度联合任务完成率作为统一指标,这一指标的工程意义在于把"安全"从"逐项合规"重新定义为"跨维度仍能完成可证明安全的任务比例"。

第 8 章会从工具栈视角整合本章定义的验证矩阵——逐项工具在 12 单元格上的覆盖广度与深度,以及当前生态在矩阵上的产业空白(尤其 cross/K)。
