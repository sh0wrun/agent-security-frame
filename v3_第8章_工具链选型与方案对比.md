# 第 8 章 工具链选型与方案对比

---

## 8.1 工具链全景与四类组织

第 6 章给出了四层防御的设计逻辑,第 7 章给出了 12 单元格的验证标准。本章把它们落实为可被工程团队选用的具体工具栈——共 28 个开源或商业工具,按用途归为四类:**评估类、事前类、事中类、事后类**。

下图是工具栈的完整全景,作为本章内容的视觉锚:

![28 个工具按 4 类组织(评估 / 事前 / 事中 / 事后)](figures/v3/tools.png)

四类的工程定位各不相同。**评估类**(9 个)对应第 7 章定义的标准化测试库与红蓝对抗框架——它们不是防护体系本身,但为防护体系的选型与持续验证提供基准载荷,典型如 R6 AEGIS 48 攻击实例集、R8 OpenClaw 系统性安全评估 205 用例、R10 ClawSafety、R11 ZeroShot、R23 AgentDojo。**事前类**(7 个)落在第 6 章 §6.1 的供应链与配置门槛上——R4 Cisco DefenseClaw 的扫描器、R18 Knownsec OpenClaw Security 的 CVE 情报订阅、R20 OpenClaw CVE Tracker 的历史漏洞回归、R5 Microsoft Agent Governance Toolkit 的智能体注册中心。**事中类**(5 个)是工具调用执行路径上的预执行防火墙、策略引擎、能力凭证解释器——R6 AEGIS、R5 AGT、R22 CaMeL、R4 DefenseClaw 的运行时沙箱、R16 ClawSafety Scanner。**事后类**(7 个)对应运营层——审计链 / 五维画像 / 信任评分 / 持续合规——R5 AGT 与 R1 奇安信内部运营组件构成主体,辅以 R2 SlowMist 的实践工具与 Splunk-OTLP 的可观测性接入。

工具栈不是 28 个工具的简单堆叠,而是按四类各自承担明确职责的"互补组合"。一个企业的智能体安全部署一般不需要全部 28 个工具,但每一类都至少需要选择一个主力工具——若评估类缺失,则防护体系无法持续被验证;若事前类缺失,则攻击面在准入前已无控;若事中类缺失,则副作用发生前无拦截;若事后类缺失,则延迟可见性问题无应对。

## 8.2 主流方案的"防护形状"对比

工具栈视角之外,工程选型还需要"方案视角"——市面上把多个工具组合为可直接部署的整体方案的厂商或开源框架,在 framework 的核心矩阵上各自呈现什么"防护形状"。本节把当前生态中六个最具代表性的方案放到核心矩阵这一统一坐标系上做对比。

下图是六个方案"防护形状"的迷你矩阵并排呈现——同一坐标系下,各方案覆盖差异一目了然:

![主流防护方案的"防护形状"迷你矩阵](figures/v3/schemes.png)

六个方案按总覆盖广度排序对比如下。

**AI-Sentinel(AI-XDR 平台)** 覆盖最广——10/12 单元格——是平台型方案的代表。它把 NDR(网络流量检测响应)与 EDR(端点检测响应)的方法论延伸到 LLM / 智能体交互层,通过流量侧全量观测、端侧运行时防护、Skill 供应链安全扫描、会话级攻击溯源、历史回溯五大能力贯穿四层。它的工程定位是"统一观测与响应面板",与下面的单体方案构成"平台 vs 单体"互补关系——单体方案在自己的主战场上深度更深,平台方案在跨层观测与溯源上面积更广。

**Microsoft Agent Governance Toolkit(R5)** 覆盖 9/12 单元格,主战场在事中层。AGT 的差异化能力是"统一策略表达"——同一套策略描述语言贯穿事前注册中心、事中策略引擎、事后审计规则、跨智能体协同握手,这是第 6 章 §6.5 跨层协同所推荐的工程参考。AGT 的工程局限在 cross/K——多智能体共享知识库的污染检测目前不在其覆盖范围内。

**AEGIS(R6)** 覆盖 6/12 单元格,主战场在事中层。AEGIS 的差异化能力是预执行防火墙 + 防篡改审计链——它把"工具调用前的判决"具象为四模块流水线(意图识别 / 策略匹配 / 异常分类 / 审计落库),并以 48 个攻击实例集作为持续回归基准。AEGIS 的工程定位是"事中拦截的标杆参考实现"——其他方案在做事中层设计时几乎都参考过 AEGIS 的模块划分。

**Cisco DefenseClaw(R4)** 覆盖 6/12 单元格,主战场在事前层。DefenseClaw 的差异化能力是 Skill 供应链扫描 + 沙箱 + 智能体间通信审计(A2A scanner)——这是当前唯一覆盖 cross/K(浅覆盖)的方案。它的"防护形状"在事前层最深、在跨智能体层呈现独特的浅覆盖——其他方案都在 cross/K 上完全空白,唯有 DefenseClaw 提供 A2A 通信的内容审计,虽不深但填补了产业空白的边角。

**Knownsec OpenClaw Security(R18)** 覆盖 4/12 单元格,主战场在事后层。它的差异化能力是 CVE 情报订阅 + 安全实践指南——把 GitHub Advisory Database 已收录的 245 个 OpenClaw 相关漏洞通过 RSS / webhook 推送到企业 SIEM,实现"准入是持续行为而非一次性事件"。它不替代事中拦截工具,但在事后运营层为情报反馈链提供持续输入。

**CaMeL(R22)** 覆盖 2/12 单元格,主战场在事中层。CaMeL 是研究工件,不应直接生产部署——但它的工程价值是"by-design 路径"的概念锚。其能力凭证 + 控制流/数据流分离机制在 R23 AgentDojo 形式化沙盒上达到 77% 可证明安全任务完成率,反向印证了 CIK 三维不可压缩性。任何后续在 in/C 与 in/K 上做深度工程的方案,都需要参考 CaMeL 的污点传播思想。

总览:六个方案叠加后,12 单元格中已有 11 格被至少一个方案覆盖。**唯一仍存在产业空白的格是 cross/K**——共享知识库污染、跨会话语义漂移、多智能体记忆传递的污染等场景下,虽有 DefenseClaw 的浅覆盖,但缺乏深度工程化的成熟方案。§8.6 会专门讨论这一空白的成因与可能的应对方向。

## 8.3 三框架分层协作架构

§8.2 的对比揭示一个工程现实——**没有任何单一方案能够覆盖所有 12 单元格**。即便是面积最广的 AI-Sentinel(10/12)也在 cross/K 与 in 层最深的能力凭证机制上留有空白。生产部署因此通常需要"多框架协作"。本节给出当前生态中较为成熟的"三框架分层协作"参考架构——以 R4 Cisco DefenseClaw + R5 Microsoft AGT + R6 AEGIS 为骨干。

三框架在四层上的分工是这样的:**事前层主力 DefenseClaw**——它的 Skill 供应链扫描器是当前生态中扫描深度最完整的工具,Skill 在被注册到 AGT 注册中心之前必须先经过 DefenseClaw 扫描。**事中层主力 AEGIS + AGT**——AEGIS 提供预执行防火墙的判决主流水线,AGT 提供统一策略表达;两者通过策略调用接口串联,AEGIS 在每次工具调用判决前向 AGT 查询当前生效策略,AGT 维护策略版本与状态。**事后层主力 AGT**——AGT 的审计链与信任评分是事后层的工程基础,DefenseClaw 与 AEGIS 的判决日志均通过标准化接口写入 AGT 审计链。**跨智能体层主力 AGT + DefenseClaw**——AGT 提供 IATP 协同握手,DefenseClaw 的 A2A scanner 提供跨智能体通信的内容审计。

三框架协作的工程关键是"接口标准化"——三方均实现第 7 章 §7.5 定义的四个验证接口,以及第 6 章 §6.5 定义的能力凭证流转协议。任何对其中某一框架的版本升级,都不应破坏另两方依赖的接口契约。这是"多框架协作"在工程实践中可持续的前提。

## 8.4 端到端集成架构与数据流

三框架协作架构进一步具象为一条端到端数据流。本节按工具调用的生命周期顺序,描述这条数据流贯穿四层时各组件的职责。

**阶段一 · Skill 注册**。新 Skill 提交到 AGT 注册中心,注册中心调用 DefenseClaw 扫描器(静态代码分析 + 依赖审计 + 签名验证 + 行为画像),Knownsec 情报订阅校验该 Skill 所引用依赖包是否在已知 CVE 列表中。任一项失败,注册被拒绝。注册成功后,Skill 进入 AGT 的白名单,被授予初始能力凭证。

**阶段二 · 工具调用判决**。智能体请求执行某 Skill 时,调用首先抵达 AEGIS 预执行防火墙。AEGIS 的意图识别模块从近期上下文提取语义意图,向 AGT 查询适用策略,与待执行调用比对。判决通过后,AEGIS 将判决结果写入 AGT 审计链,并把能力凭证递给沙箱执行环境。判决失败则中断调用链,并向事后层异常上报通道发送告警。

**阶段三 · 沙箱执行**。Skill 在沙箱中执行,执行过程被 R16 ClawSafety Scanner 监控运行时行为。若运行时行为偏离 Skill 注册阶段记录的行为画像(例如声明仅访问数据库的 Skill 突然发起外部 HTTP 请求),沙箱立即中断,异常上报至事后层。

**阶段四 · 跨智能体协同**。若 Skill 调用涉及与其他智能体协同,AGT 的 IATP 协议握手启动——双方交换 DID 凭证,DefenseClaw A2A scanner 监控握手后的会话内容。会话级凭证为双方权限交集,任何一方权限超界均触发会话隔离。

**阶段五 · 事后运营**。调用完成后,AGT 把执行结果与审计证据沉淀到防篡改审计链,五维画像系统(其中第五维"模型上下文"由 AGT 的扩展模块或 R1 奇安信内部组件提供)更新智能体长期状态。Trust Decay 信任评分根据本次调用的策略判决结果、画像偏离度、审计异常事件做动态调整。若信任分数跌破阈值,系统主动收缩该智能体权限,严重时转入隔离待查。

**阶段六 · 持续验证**。验证系统通过第 7 章 §7.5 的四个标准接口持续向防护体系发起评估请求、查询状态、读取审计、上报异常。每月一次完整红蓝对抗回归在 R23 AgentDojo 沙盒中执行,检验前述五阶段在跨层组合攻击下的整体表现。

这条数据流把 framework 的设计原则、核心矩阵、四层防御、验证矩阵全部串联为一个工程整体。它不是唯一的端到端架构——其他厂商或自研团队完全可以选择不同的工具组合——但它给出了一个"三框架协作如何串成完整链路"的可参照样本。

## 8.5 选型决策树:按规模、行业、合规

工程选型不是"选最广覆盖的方案",而是"选与企业规模、所处行业、合规要求匹配的最小可用组合"。本节给出按三个维度组织的选型决策树。

### 8.5.1 按企业规模

**小型部署(智能体 < 10 个)**:工程团队往往无独立安全运营人力,推荐"评估 + 事中"双类组合——R23 AgentDojo 提供持续验证,R5 AGT 提供最小可用的策略引擎与审计链。事前层依赖 GitHub Advisory 订阅(R18 Knownsec)而非完整扫描器,事后层依赖 AGT 内置画像而非五维完整版本。

**中型部署(智能体 10-100 个)**:推荐三框架协作架构(§8.3)——DefenseClaw + AGT + AEGIS。事前扫描器、事中预执行防火墙、事后画像与审计均独立运行,但通过标准化接口串联。这一规模通常需配 1-2 名专职智能体安全运营。

**大型部署(智能体 > 100 个)**:在三框架协作之上叠加 AI-Sentinel(AI-XDR)平台层——它的统一观测面板与会话级溯源能力是大规模运营所必需的;此规模下"个体异常检测"已不可由人工覆盖,必须依赖跨智能体的统计偏差告警。事后层的五维画像必须实现完整版本,模型上下文维度尤其关键。

### 8.5.2 按行业

**金融**:监管对身份与权限的可审计性要求极高。事前层必须强制 Skill 签名链可追溯,事中层必须强制 HITL 红线机制,事后层审计粒度必须达到判决粒度(每次策略判决的输入输出都被记录),跨智能体层 IATP 协议会话凭证必须落库。Knownsec CVE 情报订阅 + AGT + AEGIS 是该行业较常见组合。

**医疗**:HIPAA 合规要求 PHI(受保护健康信息)在工具调用中不被泄漏。事中层 in/K 单元格的注入检测召回率为关键指标,事后层 post/K 的内容指纹检测频率必须达到每日全量。CaMeL 的污点传播思想在该行业的研发团队中受关注度较高,但生产部署仍以 AGT + AEGIS 为主。

**政府 / 关键基础设施**:对部署管线投毒(S11)的应对要求最高。事前层必须实现可重现构建验证 + 签名链 + 部署清单完整性三项实践。AGT 注册中心 + DefenseClaw 是该场景的标配。

**SaaS 与互联网**:对响应延迟敏感,事中层判决延迟分位数指标(P99 < 50ms)是硬约束。AEGIS 的预执行防火墙在此场景下被广泛采用,但其策略库需要根据业务场景做针对性裁剪。

**研发与技术内部使用**:对功能保留率敏感,PinchBench 的"防御启用条件下任务完成率"是关键指标。CaMeL 在研发场景中作为"研究友好"的事中层备选。

### 8.5.3 按合规要求

**OWASP ASI 全量覆盖**:三框架协作即可达成 ASI01-ASI10 的全部覆盖。

**EU AI Act 高风险类别**:除三框架协作外,还需补充"模型行为可解释性"组件——目前生态中无开源方案,需企业自研或对接 AI-Sentinel 的会话级溯源能力。

**SOC 2 Type II**:重点在事后层审计链的完整性与不可篡改性。AGT 审计链 + WORM 存储是较为常见的实现路径。

**国内等保 / 关基**:重点在事前层的部署管线审核与跨智能体层的内容审计。R1 奇安信组件在国内合规场景下的对接深度高于国外方案。

## 8.6 cross/K 产业空白:成因与自研建议

§8.2 已经指出——六个方案叠加后,12 单元格中**仅 cross/K 仍是产业空白**。本节展开这一空白的成因与可能的工程应对方向。

### 8.6.1 空白的成因

cross/K(跨智能体 × 知识)的工程内涵是"多智能体共享知识库的污染检测与扩散阻断"。当多个智能体共享同一个 RAG 知识库或长期记忆服务时,任一智能体被污染的输出若被写入共享存储,即可在不被察觉的情况下扩散到所有引用方。这一类攻击在 R12 AIRT 中被命名为 Targeted Knowledge Base Poisoning,在第 5 章 §5.2.5 的 ClawHavoc 案例中已有完整描述。

成因之一是**问题本身工程难度高**。共享知识库的内容是渐进式累积的,污染条目可能在数十万条正常条目中潜伏数周才被某次查询触发。任何"逐条扫描"思路在大规模部署下不可行,但"统计偏差检测"又面临高误报率与高漏报率的工程困境——污染条目在统计意义上可能与某些边缘但合规的条目无法区分。

成因之二是**生态尚处于早期**。当前主流智能体部署仍以单智能体或松耦合多智能体为主,真正实现"多智能体共享知识库 + 跨会话知识传递"的大规模部署仍是少数。R12 AIRT 把这一类失效模式列为白皮书条目时使用了"emerging"一词——它是已被识别但尚未在大规模生产中频繁触发的威胁类型,因此现有厂商优先投入资源在更高频威胁上。

成因之三是**形式化基准缺失**。R23 AgentDojo 提供了单智能体场景下的形式化沙盒,但其当前版本未覆盖共享知识库污染场景。没有标准化的评估基准,工具厂商无法量化自己的方案在 cross/K 单元格上的效果——这反过来抑制了工具研发投入。

### 8.6.2 浅覆盖现状:DefenseClaw A2A Scanner

R4 Cisco DefenseClaw 的 A2A(Agent-to-Agent)scanner 是当前生态中唯一对 cross/K 做了一定深度工程化的方案。其工作原理是监控跨智能体协同会话中的内容,识别其中含有"指令性句式"或"已知污染模式"的消息。这一机制可视为一种"事中层的内容防火墙",但其覆盖深度有限——它只能识别已知模式,对新型污染或低频污染不敏感;它工作在通信层而非存储层,共享知识库内的存量污染条目不在其视野内。

A2A scanner 是产业空白边角的填补,而非空白的填充。

### 8.6.3 自研建议

对有意填补这一空白的企业研发团队,本书给出三项工程建议。

**建议一:借鉴 CaMeL 的污点传播思想做存储层扩展**。R22 CaMeL 在执行架构层定义了"控制流 vs 数据流"分离;同一思想可以延伸到存储层——共享知识库的每个条目附加来源标签(来自可信源 / 来自智能体输出 / 来自外部接入),RAG 检索时根据条目来源标签在检索结果中区分权重,智能体在使用检索结果时根据来源标签做不同的事中层判决。这种设计的工程实现成本较高,但它在思想上比"逐条扫描"更接近根本治理。

**建议二:把跨智能体知识库变更纳入第 6 章 §6.3 的五维画像**。把"模型上下文维度"从单智能体扩展为跨智能体——记录共享知识库的全量变更轨迹与各智能体的引用模式,统计偏差检测从"单智能体的画像偏移"扩展为"知识库内容分布的整体偏移"。这是事后层的延展,不需要重写事中层架构,适合作为短期工程演进路径。

**建议三:推动 R23 AgentDojo 类基准扩展共享知识库场景**。基准的缺失是工具研发投入的最大抑制因素——若有企业愿意贡献共享知识库污染的攻击载荷集合,即可推动开源社区在 AgentDojo 之上构建 cross/K 的形式化沙盒。这是工程上"用基准带动工具研发"的标准路径,适合作为产业协作的切入点。

cross/K 空白不会在短期内被任何单一厂商完全填补——它是 framework 在工程演进中留出的"待研究方向"指针,为企业研发团队和学术界共同提供了一块可投入的工程领地。

## 本章小结

工具栈不是 28 个工具的清单堆叠,而是按 4 类各自承担明确职责的互补组合;主流方案不是覆盖广度的简单竞争,而是 framework 核心矩阵上"防护形状"的差异化分布。AI-Sentinel(10/12)与 AGT(9/12)是当前生态覆盖最广的平台型与单体型代表,DefenseClaw 在事前层与跨智能体层有差异化深度,CaMeL 在事中层提供 by-design 概念锚——这五类工程哲学共同构成了 v3 framework 在工业落地上的可选项。

六方案叠加后 11/12 单元格已有覆盖,cross/K 仍是产业空白。本章给出的三项自研建议——存储层污点传播、跨智能体五维画像扩展、形式化基准扩展——为这一空白指出了短中长期的工程演进路径。第 9 章会从企业适配视角进一步展开规模 / 行业 / 合规三维下的部署模式选择,第 10 章会以真实案例横向对比三框架的反应差异。
