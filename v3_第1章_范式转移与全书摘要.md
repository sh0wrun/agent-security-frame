# 第 1 章 从模型安全到智能体安全:范式转移

---

## 1.1 AI 智能体的能力跃迁:从理解到执行

人工智能系统正在经历一次根本性的能力跃迁。传统的大语言模型(LLM)是"理解与生成"系统——接收文本输入,经过推理后产出文本输出。模型可以回答问题、分析数据,但无法直接对物理世界或数字系统产生副作用,人类阅读判断后才能转化为行动,**人类构成天然的"执行门控"**。

AI 智能体(Agentic AI)打破了这一边界。智能体将 LLM 的推理能力与外部执行能力相连——调用 Shell 命令、读写文件系统、发送邮件、浏览网页、协调其他智能体(R1 奇安信 OpenClaw 安全使用指南、R7 OpenClaw 漏洞分类法)。在这种架构下,模型的输出不再是"文本",而是"控制信号":一次工具调用可能意味着一条数据库记录的删除、一封含敏感信息邮件的发送、或一段 Shell 脚本在生产服务器上的运行。

这一跃迁可以从三个维度理解:

- **自主性维度**:传统 LLM 被动响应,智能体主动规划任务序列、自主选择工具。OWASP 将此特性总结为**代理权(Agency)**——智能体拥有的不仅是访问权限(privilege),更是自主决策与行动的权力(R3 OWASP Top 10 for Agentic Apps 2026)。
- **持久性维度**:传统 LLM 无状态,智能体具备跨会话的持久状态——长期记忆、身份配置、技能库(R7 OpenClaw 漏洞分类法)。**这意味着今天的攻击效果可能在数天后通过被污染的记忆间接触发恶意行为**。
- **连接性维度**:OpenClaw 架构通过分层 Gateway-Node-Host 设计,把 LLM 推理连接到 15+ 外部执行面(R7 OpenClaw 漏洞分类法)。每一个外部接口都是潜在攻击入口或数据出口。

**2025 年是 AI 智能体的爆发元年**;进入 2026 年,智能体加速从单点试点走向全域规模化。OpenClaw 在 2026 年 1 月重新发布后数周内 GitHub Star 突破 20 万,截至 2026 年 3 月,全球暴露在互联网的部署实例已超过 23 万(R1 奇安信 OpenClaw 安全使用指南、R20 OpenClaw CVE Tracker)。**智能体安全不再是前瞻性议题,而是此刻正在发生的、大规模工程实践问题**。

## 1.2 安全边界的本质变化:攻击面从模型扩展到全执行链

智能体能力跃迁直接导致安全问题的本质变化——**安全的核心关注点从「模型说了什么」转移到「智能体做了什么」**。

**传统 AI 安全的边界高度集中**:数据隐私、对抗样本、输出合规——这三类风险都发生在"信息处理"层面,人类判断力构成天然屏障。安全评估的核心问题是封闭的:"给定输入,模型输出是否安全?"

**智能体时代,攻击面发生了根本扩展**。R7 OpenClaw 漏洞分类法的系统性分析揭示出一个关键结构性发现:智能体的攻击面沿两个**正交的轴**展开:

- **系统轴**反映漏洞发生的架构层级。R7 将 OpenClaw 的 190 个安全公告分布在十个架构层。本书 第 4 章 在此基础上**扩展为 12 攻击面**(S1-S12,新增 S11 部署管线 + S12 流程决策)。
- **攻击轴**反映对抗技术分类:身份欺骗、策略绕过、跨层组合、提示词注入、供应链信任升级。

这两个轴的交叉意味着:**智能体安全评估不能逐漏洞打分,必须考虑漏洞间的关系**——一个架构层中的中等漏洞可能与另一层中的中等漏洞组合成完整的远程代码执行路径(R7 §5.4 SSRF→Token→Exec 三阶段链是典型例证,第 10 章 §10.2 详析)。

**风险的乘法效应**(R8 OpenClaw 系统性评估的实证):Wang 等人对 6 个 Claw 系列智能体框架的系统评估显示,**智能体化系统的整体风险显著高于底层模型单独使用时**——将 LLM 接入工具链不是风险加法,而是**风险乘法**——模型本身的脆弱性(如提示词注入)与框架的执行能力(如 Shell 执行权限)相互放大。**Agent 化 = 能力 × 攻击面**。

**CIK 维度的量化警示**(R9 Your Agent, Their Asset / CIK 实证):该研究提出 **CIK 分类法**——把智能体的持久状态归纳为能力(Capability)、身份(Identity)、知识(Knowledge)三个正交维度。在真实部署环境(连接 Gmail、Stripe、本地文件系统的 OpenClaw)中,**对任何单一维度投毒,平均攻击成功率即从基线 24.6% 跃升至 64-74%**。三个维度互不替代——任一被攻破都是灾难。CIK 是本书验证体系的根基,后续会作为 framework 的"纵轴"反复出现。

## 1.3 三个根本命题(全书理论起点)

§1.1 与 §1.2 的分析共同指向**三个根本命题**——它们既是 Agent 安全的事实陈述,也是本书 4 设计原则的直接驱动:

> **命题一**:**Agent 安全是结构性问题,不是功能性问题**。
> 它不能通过在现有系统上"添加安全功能"来解决,而需要从架构层面重新设计信任模型、策略执行机制和审计体系。
>
> **命题二**:**有效的智能体安全防护必须是预执行的**(pre-execution),而非仅仅是事后的(post-execution)。
> 智能体可在毫秒级完成数据外泄,等日志告警到达时一切已不可逆。安全控制点必须位于工具调用的执行路径上,**在副作用产生之前**完成拦截决策——"先阻断、再判断"。
>
> **命题三**:**防护体系的有效性必须是可验证的**(verifiable)。
> 「声称安全」与「证明安全」之间存在本质差距。一次性渗透测试的结论仅对测试当天的系统配置成立——验证必须是持续的、独立的、可观测的。

这三个命题之间相互独立,但共同决定了本书后续设计的轮廓——

第一个命题告诉我们,Agent 安全无法靠"在已有系统上添加一个安全模块"来解决,所以本书把"四个设计原则"当作体系的骨架而不是补丁。第二个命题决定了所有防护机制都必须能在工具调用真正发生**之前**完成判断,这一要求会反复出现在四层防御的每一节里。第三个命题要求体系自身可被独立检验,本书会专门用一章给出验证矩阵的设计。

下一节给出在三个命题之上构建的整体框架。

## 1.4 解决方案概述(本章核心)

本书把上一节的三个命题落实为一个**自上而下的四层结构**——从抽象到具体,每一层只回答一个问题:

```
顶层(为什么这样防)  ── 4 个设计原则
   │
核心矩阵(在哪防、防什么)  ── 4 层防御 × CIK 三维 = 12 单元格
   │
叠加层(防谁、用什么方案)  ── 12 攻击面 + 防护方案
   │
派生层(怎么落地)  ── 5 维画像 + 工具链 + 测试库
```

之所以不把"原则、攻击面、防御层、画像、工具"这些维度并列陈列,是因为它们承担着不同的认知角色——原则是哲学(回答"为什么"),核心矩阵是坐标(回答"在哪里"与"对什么属性"),叠加层填上具体威胁与方案(回答"防谁"),派生层是落地形式(回答"怎么做")。这一结构有两个工程价值:其一,任意维度的扩张(例如新增工具)只在派生层发生,不动摇核心矩阵;其二,任意防护工具向上可追溯到它覆盖的核心单元格、再追溯到它回应的设计原则,任意攻击面向下可定位到它需要哪些工具拦截——读者由此建立完整的心智地图。

### 1.4.1 顶层:四个设计原则

framework 的顶层是四个设计原则——**默认零信任、统一跨层策略执行、持续性安全运营、可验证的安全保障**。它们共同构成一条完整的决策链:**信任谁 → 怎么执行 → 怎么运营 → 怎么证明有效**。每条原则下还有 3-4 条更细的立论,详细展开放在第 3 章。下图是四原则与 14 条立论的卡片化全景,可作为本章的"原则一览":

### 1.4.2 核心矩阵:四层防御 × CIK 三维

framework 的核心是一个二维矩阵——横轴是按时间组织的 4 层防御,纵轴是按属性组织的 CIK 三维。

**横轴(时间维度)**——四层防御按"工具调用的时间顺序"组织:

- **事前准入**:部署前问"这个智能体或这个 Skill 可以上线吗?"
- **事中拦截**:工具调用前问"这次调用在被允许的意图范围内吗?"
- **事后运营**:调用后问"智能体的长期状态健康吗?"
- **跨智能体**:多智能体协同时问"不同智能体之间的信任关系是真实的吗?"

**纵轴(属性维度)**——CIK 三维分别对应能力(Capability)、身份(Identity)、知识(Knowledge),回答"安全机制在保护哪一类属性"。三个维度在安全机制中各自承担不同的角色:

- **C 能力维度**:管控智能体"能做什么动作"——工具调用授权、Shell 与文件系统访问、外部 API 权限、Skill 准入。安全机制在 C 维度上的产物包括 Skill 扫描器、调用前置授权策略、能力凭证(capability token)与运行时沙箱。
- **I 身份维度**:确认智能体"以谁的名义在做"——用户身份、智能体身份、子智能体身份的隔离与冒充检测。安全机制在 I 维度上的产物包括身份配置基线、Token 验证与轮换、Ring 隔离、信任评分(Trust Decay)与去中心化身份(DID)。
- **K 知识维度**:保护智能体"基于什么信息做决策"——上下文窗口、长期记忆、RAG 检索结果、系统提示词。安全机制在 K 维度上的产物包括上下文扫描、提示词注入检测、记忆隔离与防篡改、知识库污染防护。

横轴 × 纵轴得到 12 个单元格,每格独立承担一个验证单元。**事中层的三格是实时拦截主战场**(位于工具调用执行路径上,失守即产生副作用);**事后 × 知识**这一格对应记忆投毒类威胁,本书第 10 章会以邮件助手案例完整复现;**跨智能体 × 身份**这一格是多智能体协同的信任锚。下图把 CIK 三维及其在四层上的展开做了详细可视化:

![CIK 三维 + 4 层 = 12 单元格,每格的威胁/防护/指标矩阵](figures/v3/cik.png)

### 1.4.3 叠加层:12 攻击面 + 防护方案

叠加层包含两类信息——**攻击面**(攻击者视角的入口枚举)与**防护方案**(具体工具或框架的覆盖)。

**12 个攻击面 S1-S12**:S1-S10 继承自 R7 的十层攻击面分析;S11 与 S12 是本书针对部署管线投毒、流程操纵两类智能体特有失效模式所做的扩展(对应 R12 Microsoft AIRT 白皮书识别的 Provisioning Poisoning 与 Agent Flow Manipulation)。每个攻击面通过"触达 cells"映射到核心矩阵——例如"上下文窗口"这一攻击面同时触达"事中×知识"与"事后×知识"两格,因为它的威胁(直接/间接提示词注入、记忆投毒)既在事中发生,也在事后长期记忆中持续。

### 1.4.5 新一代防御方向:by-design 与形式化

近一年的智能体安全研究出现了两个值得引入本书 framework 的新方向:

- **CaMeL**(Google DeepMind / ETH Zurich,2025 年):提出"by-design"路径——把可信用户查询贡献的"控制流"与不可信工具返回的"数据流"显式分离,数据流被打 taint 标记不允许影响控制流。即使 LLM 被注入诱导生成的工具调用,仍受解释器层的能力凭证约束。这一思想**反向印证了 CIK 三维的独立性**——CaMeL 的核心机制本质上就是"强制知识不影响能力"。
- **AgentDojo**(NeurIPS 2024 D&B Track):动态沙盒环境,涵盖 workspace、banking、slack、travel 四类生活/办公环境套件。**支持"防御启用 vs 关闭"对比评估**,可输出形式化的"可证明安全任务完成率"指标。CaMeL 在该基准达到 77% 任务完成率(对比无防御 84%)的可证明安全——这是当前生态最接近"形式化验证"的工程化产物。
