# 第 9 章 部署模式与企业适配

---

## 9.1 部署形态选型:按企业规模

第 8 章已在选型决策树中按规模、行业、合规三维给出粗粒度建议。本节专门展开**规模维度**——从小型试点到大型规模化的工程演进路径,对每一档给出可直接照抄的部署蓝图。

### 9.1.1 小型部署:< 10 智能体的最小可用栈

小型部署的工程团队往往无独立安全运营人力,所有防护机制必须工程团队的开发人员就能维护。这种规模下"最小可用"远比"功能完整"重要——一个工程团队需要在一周内把基础防护栈跑起来,而非半年内追求 12 单元格全覆盖。

推荐组合:**R4 Cisco DefenseClaw + R6 AEGIS** 双栈起步。DefenseClaw 提供事前层 Skill 扫描的最低门槛,AEGIS 提供事中层预执行防火墙的标杆参考。事后层依赖 AEGIS 内置审计链而非完整画像系统,跨智能体层暂不部署专用机制——小型场景下多智能体协同尚未常态化。验证侧仅引入 R23 AgentDojo 做月度回归。

部署预算:< 1 周。这一估计基于两项工程现实——DefenseClaw 与 AEGIS 均提供 Docker 化的快速部署版本,与 OpenClaw 类主流框架的对接接口已开箱即用。一周时间含 2 天部署 / 2 天策略调优 / 1 天回归。

工程红线:即便最小栈也必须保留 HITL 红线机制(第 6 章 §6.2.6)。小型部署中"红色操作"不超过 5 个动作类型(典型如"删除生产数据库记录"、"发送外部邮件"、"调用支付 API"),全部强制人工审批,审批者由开发团队负责人直接担任。这一红线不允许"为加速演进而临时关闭"——它在小型部署的有限工程预算下提供了不依赖工具的兜底保护。

### 9.1.2 中型部署:10-100 智能体的三框架协作架构

中型部署是 framework 工程价值最直接体现的规模——团队已配备 1-2 名专职智能体安全运营,生产环境中智能体的种类与场景开始多样化,单一工具方案已无法覆盖所有场景。

推荐组合:**第 8 章 §8.3 三框架协作架构**(DefenseClaw + AGT + AEGIS)+ **R18 Knownsec OpenClaw Security**(CVE 情报订阅)+ **Splunk-OTLP**(可观测性接入)。事前层由 AGT 注册中心 + DefenseClaw 扫描器 + Knownsec 情报订阅三层叠加,事中层由 AEGIS 预执行防火墙 + AGT 统一策略引擎串联,事后层由 AGT 审计链 + 五维画像(前四维)接管,跨智能体层由 AGT IATP + DefenseClaw A2A scanner 覆盖。验证侧引入 R23 AgentDojo 月度红蓝对抗 + R6 AEGIS 48 攻击实例集每周回归。

部署预算:1-3 个月。第一个月完成基础设施部署与三框架对接,第二个月完成策略库针对企业业务的裁剪与画像基线建立,第三个月完成回归测试与运营 SOP 落地。

工程关键点是**跨框架接口契约**——三框架必须实现第 7 章 §7.5 的四个标准接口与第 6 章 §6.5 的能力凭证流转协议。运营团队的核心日常是通过 Splunk-OTLP 看板监控四层之间的能力凭证流转、信任评分曲线、画像偏离度告警。中型部署的运营成熟度直接决定该规模能否平稳演进至大型部署——若中型阶段未沉淀完整的策略库与画像基线,大型阶段的统一观测面板会失去基础。

### 9.1.3 大型部署:> 100 智能体的全栈 + 平台化

大型部署是真正考验 framework 跨层一致性的规模——单个智能体的异常已不可由人工逐项审视,必须依赖跨智能体的统计偏差告警与平台化的统一观测。

推荐组合:**三框架协作架构 + AI-Sentinel(AI-XDR 平台层) + Kata Containers(强沙箱) + 独立验证系统 + SOC 2 SIEM**。AI-Sentinel 在三框架之上提供统一观测与会话级溯源能力,Kata Containers 把事中层沙箱从容器级隔离升级为虚拟机级隔离(应对 R7 漏洞分类法揭示的容器逃逸类高风险),独立验证系统通过第 7 章 §7.5 的四接口持续运行(物理上与防护体系隔离部署,人员权限互不重叠),SIEM 集成把全部审计事件、信任评分变化、画像告警接入企业的安全事件响应平台。

部署预算:6-12 个月。前 3 个月完成基础设施扩容与平台层对接,中间 3-6 个月完成全栈策略库 / 画像基线 / 验证基准的深度调优,最后 3 个月完成 SOC 2 / EU AI Act 等合规对接与运营成熟度评估。这一节奏符合大型企业的工程治理常规——年度安全规划与季度演练形成闭环。

**高敏感链路的额外加固**:对涉及关键数据 / 大额资金 / 敏感客户的智能体链路,在前述全栈基础上额外加挂"R22 CaMeL 设计原理 + R23 AgentDojo 形式化基准"。这里的"加挂"并非生产部署 CaMeL 解释器(其侵入性过强),而是把 CaMeL 的污点传播思想固化到 AGT 策略库的扩展规则中,并把 AgentDojo 的"可证明安全任务完成率"作为该链路上线前的硬指标——任务完成率必须 > 75%,否则该链路不允许接入生产。

## 9.2 行业垂直适配

不同行业对智能体安全的诉求差异源自三个因素:监管强度、数据敏感度、容错预算。本节按五个典型行业各自的工程关键点展开。

### 9.2.1 金融

金融行业的核心诉求是**身份与权限的可审计性**。任何一次工具调用必须能追溯到完整的"用户 → 业务系统 → 智能体 → Skill"链路,任一环节的身份模糊都意味着合规风险。

事前层必须强制 Skill 签名链可追溯——第 6 章 §6.1.3 的可重现构建验证、签名链可追溯、部署清单完整性三项实践全部为硬约束。事中层必须强制 HITL 红线机制——交易类、账户变更类、敏感数据导出类动作全部进入人工审批通道,审批负载控制(同一审批者每日待审上限)与审批上下文最小化是合规审计的重点。事后层审计粒度必须达到**判决粒度**(第 6 章 §6.3.2)——每次策略判决的输入、输出、依据均落库,WORM 存储保留期不少于 7 年。跨智能体层 IATP 协议会话凭证必须落库,任何会话权限超界事件即触发监管报告。

典型工具组合:**Knownsec 情报订阅 + AGT + AEGIS** 是金融行业较常见组合。Knownsec 在国内金融场景下与 SIEM 的对接深度高于国外方案,AGT 的统一策略表达便于满足"策略可审计"要求,AEGIS 的判决延迟分位数指标(P99 < 50ms)在交易场景中至关重要。

### 9.2.2 医疗

医疗行业的核心诉求是 **PHI(受保护健康信息)在工具调用中不被泄漏**。HIPAA 合规要求 PHI 的访问、使用、传输全部可追踪,任何未经授权的 PHI 出现在工具调用参数或响应中即构成事件。

事中层 in/K(事中 × 知识)单元格的注入检测召回率为关键指标——智能体在引用包含 PHI 的上下文时,事中层必须确保该上下文不会被诱导泄漏到外部接口。事后层 post/K 的内容指纹检测频率必须达到**每日全量**——长期记忆与 RAG 知识库中是否有 PHI 残留是日常审计重点。

R22 CaMeL 的污点传播思想在医疗行业的研发团队中受关注度较高——把 PHI 数据流标记为 taint 来源,任何工具调用控制流不允许由 PHI 数据流影响,这一思想在医疗场景下契合度极高。但生产部署仍以 R5 AGT + R6 AEGIS 为主——CaMeL 的解释器侵入性在医疗系统的高合规要求下尚未取得普遍认可。

### 9.2.3 政府与关键基础设施

政府与关键基础设施场景的核心诉求是**对部署管线投毒(S11)的应对**。这一类场景的供应链来自多家厂商,任一环节被污染都可能影响关键系统稳定性。

事前层必须实现可重现构建验证 + 签名链 + 部署清单完整性三项实践,且这三项必须由独立第三方验证——企业自身的 CI/CD 系统不被视为可信验证源。AGT 注册中心 + DefenseClaw 扫描器是该场景的标配,部署清单的完整性核验由独立的合规审计机构每季度核查。

国内等保与关基场景下,**R1 奇安信组件**的对接深度高于国外方案——内部运营经验与国内合规要求高度契合,跨智能体层的 A2A 内容审计也在 R1 内部组件中有针对性扩展。

### 9.2.4 SaaS 与互联网

SaaS 与互联网场景的核心诉求是**响应延迟敏感**——事中层判决延迟分位数(P99 < 50ms)是硬约束,任何拖慢用户感知的防护机制都会被产品团队反推删除。

R6 AEGIS 的预执行防火墙在该场景下被广泛采用,但其策略库需要根据业务场景做针对性裁剪——通用策略库的判决延迟通常超过 100ms,裁剪后才能进入 50ms 区间。事中层的判决依据缓存与意图识别模型的轻量化是该场景下的工程优化重点。

事后层在该场景下偏向**统计偏差告警而非逐项画像**——单用户级别的画像在大流量下不可行,必须按业务维度(如按租户 / 按场景 / 按产品线)聚合。AI-Sentinel 平台层在 SaaS 场景的接入价值最高——它的会话级溯源能力是大流量下"事后定位异常"的关键工程能力。

### 9.2.5 研发与技术内部使用

研发与技术内部场景的核心诉求是**功能保留率**——智能体被工程团队用于代码生成、调试、文档撰写,任何"为安全而限制功能"的策略都会被快速绕过或停用。

PinchBench 的"防御启用条件下任务完成率"是该场景的关键指标——典型阈值为 > 90%,即防护机制不应使任务完成率下降超过 10%。R22 CaMeL 在研发场景中作为"研究友好"的事中层备选——研发团队对 CaMeL 解释器的侵入性容忍度高于其他场景,且其控制流分离思想在代码生成与调试任务中有天然契合度。

## 9.3 合规与法规映射

### 9.3.1 OWASP Top 10 for Agentic Applications

R3 OWASP ASI 是当前最具产业共识的智能体安全风险分类法,ASI01-ASI10 十类风险与 framework 各层防御机制的对应已在第 5 章 §5.4.1 给出完整映射。**三框架协作架构**(第 8 章 §8.3)在 ASI01-ASI10 上即可达成全量覆盖,无需补充额外组件。OWASP ASI 是其他更严格合规框架(EU AI Act、SOC 2)的工程基础,所有 v3 framework 部署都默认包含 OWASP ASI 的覆盖。

### 9.3.2 EU AI Act:高风险类别

EU AI Act 把智能体应用按风险等级分类,涉及关键基础设施、教育评估、就业筛选、执法、移民管理等场景的应用归为"高风险类别"。这一类别要求"模型行为可解释性"——监管机构有权要求企业说明任意一次智能体决策的依据。

framework 在这一要求上的工程对接需要补充"模型行为可解释性"组件——目前生态中无开源方案。企业研发团队需要自研或对接 AI-Sentinel 平台层的会话级溯源能力。AI-Sentinel 的 Kill Chain 还原功能可以把一次智能体决策追溯到完整的"上下文 → 工具调用 → 输出"链路,这一能力在 EU AI Act 监管对接中具有高度合规价值。

### 9.3.3 HIPAA(美国医疗合规)

HIPAA 要求 PHI 的访问、使用、传输全部可追踪。framework 的工程对接重点在事后层审计粒度与事中层 in/K 注入检测——已在 §9.2.2 详述。

### 9.3.4 SOC 2 Type II

SOC 2 Type II 的重点在事后层审计链的**完整性与不可篡改性**。AGT 审计链 + WORM(write once read many)存储是较为常见的实现路径。审计事件的写入必须签名链完整、保留期符合行业要求(典型为 7 年)、独立第三方可读但不可改。验证系统通过第 7 章 §7.5 §接口三 · 审计读取接口实现"独立第三方可读"的要求——其只读权限本身即被审计链记录,符合 SOC 2 的双向审计要求。

### 9.3.5 国内等保与关基

国内等保 / 关基的工程对接重点在事前层的部署管线审核与跨智能体层的内容审计。R1 奇安信组件在国内合规场景下的对接深度高于国外方案——内部经验、合规接口、监管对接均已在多个企业落地。R1 + AGT + AEGIS 的组合在国内大型部署中较为常见。

## 9.4 CLAW-10 企业就绪度评估

R21 CLAW-10 是 v3 framework 推荐采用的企业就绪度评估清单。它把"企业是否具备运行智能体安全防护体系的工程能力"分解为十项可量化检查项,每项满分 10 分,总分 100 分。本节给出十项检查项的具体定义。

**项 1 · 资产清晰度**:企业是否对所有智能体、Skill、数据流、外部接口建立了完整清单?清单更新延迟是否 ≤ 24 小时?

**项 2 · 身份治理深度**:智能体身份配置是否实现了第 6 章 §6.1.4 的基线 + 异常使用检测 + 信任评分三层闭环?子智能体派生关系是否可追溯?

**项 3 · 准入门槛严格度**:Skill 注册前是否经过扫描器、签名链、CVE 情报三层检查?未通过的 Skill 是否被注册中心硬拒?

**项 4 · 事中拦截覆盖度**:事中层判决延迟分位数(P99)是否在业务可容忍范围?判决拒绝率与误拒率是否有持续监控?

**项 5 · HITL 红线完备度**:红色操作清单是否完整?审批负载控制、审批上下文最小化、异常审批路径三项工程实践是否落地?

**项 6 · 审计链完整性**:审计粒度是否达到判决粒度?WORM 存储保留期是否符合合规要求?审计链签名链验证是否每日执行?

**项 7 · 五维画像运营**:Skill / 行为 / 权限 / 账号设备 / 模型上下文五个维度是否全部上线?画像偏离度告警的准确率是否经过持续校准?

**项 8 · Trust Decay 工程化**:信任评分四级响应(绿/黄/橙/红)是否完整实现?跨日期累积约束是否被信任评分系统耦合?

**项 9 · 跨智能体协同治理**:DID + IATP 协议是否实现?智能体冒充检测的双层身份核验(密码学身份 + 行为指纹)是否同时运行?

**项 10 · 持续验证体系**:第 7 章 §7.3 八测试集合 + PinchBench 是否定期回归?红蓝对抗回归是否每月执行?四个标准化验证接口是否完整实现?

CLAW-10 的工程价值不在于"满分 100 分",而在于让企业认清自己当前所处的演进阶段——典型的小型部署 CLAW-10 总分通常在 40-60 区间,中型部署在 60-80 区间,大型部署应当达到 85+ 区间。低于 40 分意味着企业的智能体部署存在结构性安全风险,应当暂缓上线扩展并优先补齐基础栈。

## 9.5 企业落地路线图:分阶段演进

最后一节给出企业从零开始落地 v3 framework 的分阶段路线图。这一路线图不是固定模板,而是按工程现实对小型 → 中型 → 大型演进做的分阶段抽象,具体节奏可按企业资源做调整。

**阶段一(0-1 个月)· 最小可用栈**:DefenseClaw + AEGIS 双栈起步,5 个红色操作 HITL 红线落地,事中判决延迟基线建立。这一阶段的目标是"让防护机制对生产工作流不构成阻塞"——在工程团队能接受的延迟预算内,把基础拦截能力跑起来。

**阶段二(1-3 个月)· 三框架协作**:加入 AGT 统一策略表达 + Knownsec CVE 情报订阅 + Splunk-OTLP 可观测性接入。这一阶段的目标是"实现跨层一致性"——能力凭证在四层之间流转无间隙,策略变更同步生效不存在窗口。

**阶段三(3-6 个月)· 五维画像 + Trust Decay**:事后层从"基础审计"升级为"持续运营",五维画像全部上线,Trust Decay 四级响应工程化。这一阶段的目标是"让运营从被动响应转为主动监测"——每日工作日终前所有橙色与红色告警必须有处置记录。

**阶段四(6-12 个月)· 全栈 + 平台化**:加入 AI-Sentinel(或同类 AI-XDR 平台)+ Kata Containers 强沙箱 + 独立验证系统 + SIEM 集成。这一阶段的目标是"实现规模化运营"——单智能体级别的异常检测让位于跨智能体的统计偏差告警,运营人力从"逐项盯告警"升级为"系统性调优策略库"。

**阶段五(12-18 个月)· 高敏感链路加固 + 形式化基准**:在全栈基础上对高敏感链路加挂 CaMeL 设计原理 + AgentDojo 形式化基准,推动"可证明安全任务完成率"作为关键链路上线的硬指标。这一阶段的目标是"在 framework 上探索新一代防御方向"——by-design 路径的工程化经验沉淀,为下一代生产部署做储备。

每一阶段在结束时执行一次 CLAW-10 评估并对照预期分数。若实际分数低于预期 10 分以上,应当暂缓推进下一阶段,先补齐欠缺项。这一约束保证企业演进不在"功能全但运营薄弱"的状态下扩张——后者是大型部署中最常见的故障源。

## 本章小结

部署模式选型的核心立场是"匹配规模而非追求广度"——小型部署的工程预算决定了它必须以"最小可用"为目标,大型部署的运营复杂度则要求"全栈 + 平台化"。本章给出的三档蓝图与五个行业的垂直适配,其工程价值不在于固化模板,而在于让工程团队认清自己所处的演进阶段与下一阶段需要的工程投入。

CLAW-10 评估清单与五阶段路线图共同构成 framework 在企业落地上的工程治理工具——评估清单识别"现在在哪",路线图指出"下一步该往哪走"。两者结合使企业的智能体安全演进可以被量化、可被审计、可被持续治理。

第 10 章会以四个真实案例完整展示前九章建立的 framework 在攻防中的具体反应——案例既是 framework 自洽性的最终工程证据,也是企业落地阶段红队基准的素材库。
